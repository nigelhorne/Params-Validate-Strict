   83: =head2 validate_strict
   84: 
   85: Validates a set of parameters against a schema.
   86: 
   87: This function takes two mandatory arguments:
   88: 
   89: =over 4
   90: 
   91: =item * C<schema> || C<members>
   92: 
   93: A reference to a hash that defines the validation rules for each parameter.
   94: The keys of the hash are the parameter names, and the values are either a string representing the parameter type or a reference to a hash containing more detailed rules.
   95: 
   96: For some sort of compatibility with L<Data::Processor>,
   97: it is possible to wrap the schema within a hash like this:
   98: 
   99:   $schema = {
  100:     description => 'Describe what this schema does',
  101:     error_msg => 'An error message',
  102:     schema => {
  103:       # ... schema goes here
  104:     }
  105:   }
  106: 
  107: =item * C<args> || C<input>
  108: 
  109: A reference to a hash containing the parameters to be validated.
  110: The keys of the hash are the parameter names, and the values are the parameter values.
  111: 
  112: =back
  113: 
  114: It takes optional arguments:
  115: 
  116: =over 4
  117: 
  118: =item * C<description>
  119: 
  120: What the schema does,
  121: used in error messages.
  122: 
  123: =item * C<error_msg>
  124: 
  125: Overrides the default message when something doesn't validate.
  126: 
  127: =item * C<unknown_parameter_handler>
  128: 
  129: This parameter describes what to do when a parameter is given that is not in the schema of valid parameters.
  130: It must be one of C<die>, C<warn>, or C<ignore>.
  131: 
  132: It defaults to C<die> unless C<carp_on_warn> is given, in which case it defaults to C<warn>.
  133: 
  134: =item * C<logger>
  135: 
  136: A logging object that understands messages such as C<error> and C<warn>.
  137: 
  138: =item * C<custom_types>
  139: 
  140: A reference to a hash that defines reusable custom types.
  141: Custom types allow you to define validation rules once and reuse them throughout your schema,
  142: making your validation logic more maintainable and readable.
  143: 
  144: Each custom type is defined as a hash reference containing the same validation rules available for regular parameters
  145: (C<type>, C<min>, C<max>, C<matches>, C<memberof>, C<enum>, C<notmemberof>, C<callback>, etc.).
  146: 
  147:   my $custom_types = {
  148:     email => {
  149:       type => 'string',
  150:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/,
  151:       error_msg => 'Invalid email address format'
  152:     }, phone => {
  153:       type => 'string',
  154:       matches => qr/^\+?[1-9]\d{1,14}$/,
  155:       min => 10,
  156:       max => 15
  157:     }, percentage => {
  158:       type => 'number',
  159:       min => 0,
  160:       max => 100
  161:     }, status => {
  162:       type => 'string',
  163:       memberof => ['draft', 'published', 'archived']
  164:     }
  165:   };
  166: 
  167:   my $schema = {
  168:     user_email => { type => 'email' },
  169:     contact_number => { type => 'phone', optional => 1 },
  170:     completion => { type => 'percentage' },
  171:     post_status => { type => 'status' }
  172:   };
  173: 
  174:   my $validated = validate_strict(
  175:     schema => $schema,
  176:     input => $input,
  177:     custom_types => $custom_types
  178:   );
  179: 
  180: Custom types can be extended or overridden in the schema by specifying additional constraints:
  181: 
  182:   my $schema = {
  183:     admin_username => {
  184:       type => 'username',  # Uses custom type definition
  185:       min => 5,            # Overrides custom type's min value
  186:       max => 15            # Overrides custom type's max value
  187:     }
  188:   };
  189: 
  190: Custom types work seamlessly with nested schema, optional parameters, and all other validation features.
  191: 
  192: =back
  193: 
  194: The schema can define the following rules for each parameter:
  195: 
  196: =over 4
  197: 
  198: =item * C<type>
  199: 
  200: The data type of the parameter.
  201: Valid types are C<string>, C<integer>, C<number>, C<float> C<boolean>, C<hashref>, C<arrayref>, C<object> and C<coderef>.
  202: 
  203: A type can be an arrayref when a parameter could have different types (e.g. a string or an object).
  204: 
  205:   $schema = {
  206:     username => [
  207:       { type => 'string', min => 3, max => 50 },	# Name
  208:       { type => 'integer', 'min' => 1 },	# UID that isn't root
  209:     ]
  210:   };
  211: 
  212: As a shorthand, C<type> itself may be an arrayref of type name strings (a I<union type>)
  213: when all other constraints are shared between the alternatives:
  214: 
  215:   $schema = {
  216:     data => { type => ['string', 'arrayref'] },
  217:     id   => { type => ['string', 'integer'], optional => 1 },
  218:   };
  219: 
  220: This is equivalent to the full array-of-rules form but more concise.
  221: Every other key in the rule hash (C<optional>, C<min>, C<max>, C<matches>, etc.)
  222: is inherited by each candidate type and validated independently against it.
  223: Type names are tried left-to-right; the first match wins and its coercion
  224: (e.g. numeric types) is propagated back to the caller.
  225: If the value fails all candidate types, validation croaks with a message
  226: listing the union members.
  227: 
  228: =item * C<can>
  229: 
  230: The parameter must be an object that understands the method C<can>.
  231: C<can> can be a simple scalar string of a method name,
  232: or an arrayref of a list of method names, all of which must be supported by the object.
  233: 
  234:    $schema = {
  235:      gedcom => { type => object, can => 'get_individual' }
  236:    }
  237: 
  238: =item * C<isa>
  239: 
  240: The parameter must be an object of type C<isa>.
  241: 
  242: =item * C<memberof>
  243: 
  244: The parameter must be a member of the given arrayref.
  245: 
  246:   status => {
  247:     type => 'string',
  248:     memberof => ['draft', 'published', 'archived']
  249:   }
  250: 
  251:   priority => {
  252:     type => 'integer',
  253:     memberof => [1, 2, 3, 4, 5]
  254:   }
  255: 
  256: For string types, the comparison is case-sensitive by default. Use the C<case_sensitive>
  257: flag to control this behavior:
  258: 
  259:   # Case-sensitive (default) - must be exact match
  260:   code => {
  261:     type => 'string',
  262:     memberof => ['ABC', 'DEF', 'GHI']
  263:     # 'abc' will fail
  264:   }
  265: 
  266:   # Case-insensitive - any case accepted
  267:   code => {
  268:     type => 'string',
  269:     memberof => ['ABC', 'DEF', 'GHI'],
  270:     case_sensitive => 0
  271:     # 'abc', 'Abc', 'ABC' all pass, original case preserved
  272:   }
  273: 
  274: For numeric types (C<integer>, C<number>, C<float>), the comparison uses numeric
  275: equality (C<==> operator):
  276: 
  277:   rating => {
  278:     type => 'number',
  279:     memberof => [0.5, 1.0, 1.5, 2.0]
  280:   }
  281: 
  282: Note that C<memberof> cannot be combined with C<min> or C<max> constraints as they
  283: serve conflicting purposes - C<memberof> defines an explicit whitelist while C<min>/C<max>
  284: define ranges.
  285: 
  286: =item * C<enum>
  287: 
  288: Same as C<memberof>.
  289: 
  290: =item * C<notmemberof>
  291: 
  292: The parameter must not be a member of the given arrayref (blacklist).
  293: This is the inverse of C<memberof>.
  294: 
  295:   username => {
  296:     type => 'string',
  297:     notmemberof => ['admin', 'root', 'system', 'administrator']
  298:   }
  299: 
  300:   port => {
  301:     type => 'integer',
  302:     notmemberof => [22, 23, 25, 80, 443]  # Reserved ports
  303:   }
  304: 
  305: Like C<memberof>, string comparisons are case-sensitive by default but can be controlled
  306: with the C<case_sensitive> flag:
  307: 
  308:   # Case-sensitive (default)
  309:   username => {
  310:     type => 'string',
  311:     notmemberof => ['Admin', 'Root']
  312:     # 'admin' would pass, 'Admin' would fail
  313:   }
  314: 
  315:   # Case-insensitive
  316:   username => {
  317:     type => 'string',
  318:     notmemberof => ['Admin', 'Root'],
  319:     case_sensitive => 0
  320:     # 'admin', 'ADMIN', 'Admin' all fail
  321:   }
  322: 
  323: The blacklist is checked after any C<transform> rules are applied, allowing you to
  324: normalize input before checking:
  325: 
  326:   username => {
  327:     type => 'string',
  328:     transform => sub { lc($_[0]) },  # Normalize to lowercase
  329:     notmemberof => ['admin', 'root', 'system']
  330:   }
  331: 
  332: C<notmemberof> can be combined with other validation rules:
  333: 
  334:   username => {
  335:     type => 'string',
  336:     notmemberof => ['admin', 'root', 'system'],
  337:     min => 3,
  338:     max => 20,
  339:     matches => qr/^[a-z0-9_]+$/
  340:   }
  341: 
  342: =item * C<case_sensitive>
  343: 
  344: A boolean value indicating whether string comparisons should be case-sensitive.
  345: This flag affects the C<memberof> and C<notmemberof> validation rules.
  346: The default value is C<1> (case-sensitive).
  347: 
  348: When set to C<0>, string comparisons are performed case-insensitively, allowing values
  349: with different casing to match. The original case of the input value is preserved in
  350: the validated output.
  351: 
  352:   # Case-sensitive (default)
  353:   status => {
  354:     type => 'string',
  355:     memberof => ['Draft', 'Published', 'Archived'] # Input 'draft' will fail - must match exact case
  356:   }
  357: 
  358:   # Case-insensitive
  359:   status => {
  360:     type => 'string',
  361:     memberof => ['Draft', 'Published', 'Archived'],
  362:     case_sensitive => 0 # Input 'draft', 'DRAFT', or 'DrAfT' will all pass
  363:   }
  364: 
  365:   country_code => {
  366:     type => 'string',
  367:     memberof => ['US', 'UK', 'CA', 'FR'],
  368:     case_sensitive => 0  # Accept 'us', 'US', 'Us', etc.
  369:   }
  370: 
  371: This flag has no effect on numeric types (C<integer>, C<number>, C<float>) as numbers
  372: do not have case.
  373: 
  374: =item * C<min>/C<minimum>
  375: 
  376: The minimum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  377: 
  378: =item * C<max>
  379: 
  380: The maximum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  381: 
  382: =item * C<matches>
  383: 
  384: A regular expression that the parameter value must match.
  385: Checks all members of arrayrefs.
  386: 
  387: =item * C<nomatch>
  388: 
  389: A regular expression that the parameter value must not match.
  390: Checks all members of arrayrefs.
  391: 
  392: =item * C<position>
  393: 
  394: For routines and methods that take positional args,
  395: this integer value defines which position the argument will be in.
  396: If this is set for all arguments,
  397: C<validate_strict> will return a reference to an array, rather than a reference to a hash.
  398: 
  399: =item * C<regex>
  400: 
  401: Synonym of matches
  402: 
  403: =item * C<description>
  404: 
  405: The description of the rule
  406: 
  407: =item * C<callback>
  408: 
  409: A code reference to a subroutine that performs custom validation logic.
  410: The subroutine should accept the parameter value, the argument list and the schema as arguments and return true if the value is valid, false otherwise.
  411: 
  412: Use this to test more complex examples:
  413: 
  414:   my $schema = {
  415:     even_number => {
  416:       type => 'integer',
  417:       callback => sub { $_[0] % 2 == 0 }
  418:   };
  419: 
  420:   # Specify the arguments for a routine which has a second, optional argument, which, if given, must be less than or equal to the first
  421:   my $schema = {
  422:     first => {
  423:       type => 'integer'
  424:     }, second => {
  425:       type => 'integer',
  426:       optional => 1,
  427:       callback => sub {
  428:         my($value, $args) = @_;
  429: 	# The 'defined' is needed in case 'second' is evaluated before 'first'
  430: 	return (defined($args->{first}) && $value <= $args->{first}) ? 1 : 0
  431:       }
  432:     }
  433:   };
  434: 
  435: =item * C<optional>
  436: 
  437: A boolean value indicating whether the parameter is optional.
  438: If true, the parameter is not required.
  439: If false or omitted, the parameter is required.
  440: 
  441: It can be a reference to a code snippet that will return true or false,
  442: to determine if the parameter is optional or not.
  443: The code will be called with two arguments: the value of the parameter and hash ref of all parameters:
  444: 
  445:   my $schema = {
  446:     optional_field => {
  447:       type => 'string',
  448:       optional => sub {
  449:         my ($value, $all_params) = @_;
  450:         return $all_params->{make_optional} ? 1 : 0;
  451:       }
  452:     },
  453:     make_optional => { type => 'boolean' }
  454:   };
  455: 
  456:   my $result = validate_strict(schema => $schema, input => { make_optional => 1 });
  457: 
  458: =item * C<default>
  459: 
  460: Populate missing optional parameters with the specified value.
  461: Note that this value is not validated.
  462: 
  463:   username => {
  464:     type => 'string',
  465:     optional => 1,
  466:     default => 'guest'
  467:   }

  491:

  496: =item * C<schema>
  497: 
  498: You can validate nested hashrefs and arrayrefs using the C<schema> property:

  500:     my $schema = {
  501:         user => {	# 'user' is a hashref
  502:             type => 'hashref',
  503:             schema => {	# Specify what the elements of the hash should be

  514:             schema => {

  516:                 tags => {
  517:                     type => 'arrayref',

  521:                     }
  522:                 }
  523:             }
  524:         }
  525:     };
  526: 
  527: =item * C<validate>
  528: 
  529: A snippet of code that validates the input.
  530: It's passed the input arguments,
  531: and return a string containing a reason for rejection,
  532: or undef if it's allowed.
  533: 
  534:     my $schema = {
  535:       user => {
  536:         type => 'string',
  537: 	validate => sub {
  538: 	  if($_[0]->{'password'} eq 'bar') {
  539: 	    return undef;
  540: 	  }
  541: 	  return 'Invalid password, try again';
  542: 	}
  543:       }, password => {
  544:          type => 'string'
  545:       }
  546:     };

  548: =item * C<transform>
  549: 
  550: A code reference to a subroutine that transforms/sanitizes the parameter value before validation.
  551: The subroutine should accept the parameter value as an argument and return the transformed value.
  552: The transformation is applied before any validation rules are checked, allowing you to normalize
  553: or clean data before it is validated.
  554: 
  555: Common use cases include trimming whitespace, normalizing case, formatting phone numbers,
  556: sanitizing user input, and converting between data formats.
  557: 
  558:   # Simple string transformations
  559:   username => {
  560:     type => 'string',
  561:     transform => sub { lc(trim($_[0])) },  # lowercase and trim
  562:     matches => qr/^[a-z0-9_]+$/
  563:   }
  564: 
  565:   email => {
  566:     type => 'string',
  567:     transform => sub { lc(trim($_[0])) },  # normalize email
  568:     matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  569:   }
  570: 
  571:   # Array transformations
  572:   tags => {
  573:     type => 'arrayref',
  574:     transform => sub { [map { lc($_) } @{$_[0]}] },  # lowercase all elements
  575:     element_type => 'string'
  576:   }
  577: 
  578:   keywords => {
  579:     type => 'arrayref',
  580:     transform => sub {
  581:       my @arr = map { lc(trim($_)) } @{$_[0]};
  582:       my %seen;
  583:       return [grep { !$seen{$_}++ } @arr];  # remove duplicates
  584:     }
  585:   }
  586: 
  587:   # Numeric transformations
  588:   quantity => {
  589:     type => 'integer',
  590:     transform => sub { int($_[0] + 0.5) },  # round to nearest integer
  591:     min => 1
  592:   }
  593: 
  594:   # Sanitization
  595:   slug => {
  596:     type => 'string',
  597:     transform => sub {
  598:       my $str = lc(trim($_[0]));
  599:       $str =~ s/[^\w\s-]//g;  # remove special characters
  600:       $str =~ s/\s+/-/g;      # replace spaces with hyphens
  601:       return $str;
  602:     },
  603:     matches => qr/^[a-z0-9-]+$/
  604:   }
  605: 
  606:   phone => {
  607:     type => 'string',
  608:     transform => sub {
  609:       my $str = $_[0];
  610:       $str =~ s/\D//g;  # remove all non-digits
  611:       return $str;
  612:     },
  613:     matches => qr/^\d{10}$/
  614:   }
  615: 
  616: The C<transform> function is applied to the value before any validation checks (C<min>/C<minimum>, C<max>,
  617: C<matches>, C<callback>, etc.), ensuring that validation rules are checked against the cleaned data.
  618: 
  619: Transformations work with all parameter types including nested structures:
  620: 
  621:   user => {
  622:     type => 'hashref',
  623:     schema => {
  624:       name => {
  625:         type => 'string',
  626:         transform => sub { trim($_[0]) }
  627:       },
  628:       email => {
  629:         type => 'string',
  630:         transform => sub { lc(trim($_[0])) }
  631:       }
  632:     }
  633:   }
  634: 
  635: Transformations can also be defined in custom types for reusability:
  636: 
  637:   my $custom_types = {
  638:     email => {
  639:       type => 'string',
  640:       transform => sub { lc(trim($_[0])) },
  641:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  642:     }
  643:   };
  644: 
  645: Note that the transformed value is what gets returned in the validated result and is what
  646: subsequent validation rules will check against. If a transformation might fail, ensure it
  647: handles edge cases appropriately.
  648: It is the responsibility of the transformer to ensure that the type of the returned value is correct,
  649: since that is what will be validated.
  650:

  652:

  663: =item * C<validator>
  664: 
  665: A synonym of C<validate>, for compatibility with L<Data::Processor>.

  667: =item * C<cross_validation>
  668: 
  669: A reference to a hash that defines validation rules that depend on more than one parameter.
  670: Cross-field validations are performed after all individual parameter validations have passed,
  671: allowing you to enforce business logic that requires checking relationships between different fields.
  672: 
  673: Each cross-validation rule is a key-value pair where the key is a descriptive name for the validation
  674: and the value is a code reference that accepts a hash reference of all validated parameters.
  675: The subroutine should return C<undef> if the validation passes, or an error message string if it fails.
  676: 
  677:   my $schema = {
  678:     password => { type => 'string', min => 8 },
  679:     password_confirm => { type => 'string' }
  680:   };
  681: 
  682:   my $cross_validation = {
  683:     passwords_match => sub {
  684:       my $params = shift;
  685:       return $params->{password} eq $params->{password_confirm}
  686:         ? undef : "Passwords don't match";
  687:     }
  688:   };
  689: 
  690:   my $validated = validate_strict(
  691:     schema => $schema,
  692:     input => $input,
  693:     cross_validation => $cross_validation
  694:   );
  695: 
  696: Common use cases include password confirmation, date range validation, numeric comparisons,
  697: and conditional requirements:
  698: 
  699:   # Date range validation
  700:   my $cross_validation = {
  701:     date_range_valid => sub {
  702:       my $params = shift;
  703:       return $params->{start_date} le $params->{end_date}
  704:         ? undef : "Start date must be before or equal to end date";
  705:     }
  706:   };
  707: 
  708:   # Price range validation
  709:   my $cross_validation = {
  710:     price_range_valid => sub {
  711:       my $params = shift;
  712:       return $params->{min_price} <= $params->{max_price}
  713:         ? undef : "Minimum price must be less than or equal to maximum price";
  714:     }
  715:   };
  716: 
  717:   # Conditional required field
  718:   my $cross_validation = {
  719:     address_required_for_delivery => sub {
  720:       my $params = shift;
  721:       if ($params->{shipping_method} eq 'delivery' && !$params->{delivery_address}) {
  722:         return "Delivery address is required when shipping method is 'delivery'";
  723:       }
  724:       return undef;
  725:     }
  726:   };

  728: Multiple cross-validations can be defined in the same hash, and they are all checked in order.
  729: If any cross-validation fails, the function will C<croak> with the error message returned by the validation:
  730: 
  731:   my $cross_validation = {
  732:     passwords_match => sub {
  733:       my $params = shift;
  734:       return $params->{password} eq $params->{password_confirm}
  735:         ? undef : "Passwords don't match";
  736:     },
  737:     emails_match => sub {
  738:       my $params = shift;
  739:       return $params->{email} eq $params->{email_confirm}
  740:         ? undef : "Email addresses don't match";
  741:     },

  743:       my $params = shift;
  744:       my $current_year = (localtime)[5] + 1900;

  748:     }
  749:   };

  751: Cross-validations receive the parameters after individual validation and transformation have been applied,
  752: so you can rely on the data being in the correct format and type:
  753: 
  754:   my $schema = {
  755:     email => {

  797: =item * C<relationships>
  798: 
  799: A reference to an array that defines validation rules based on relationships between parameters.
  800: Relationship validations are performed after all individual parameter validations have passed,
  801: but before cross-validations.
  802: 
  803: Each relationship is a hash reference with a C<type> field and additional fields depending on the type:
  804: 
  805: =over 4
  806: 
  807: =item * B<mutually_exclusive>
  808: 
  809: Parameters that cannot be specified together.
  810: 
  811:   relationships => [
  812:     {
  813:       type => 'mutually_exclusive',
  814:       params => ['file', 'content'],
  815:       description => 'Cannot specify both file and content'
  816:     }
  817:   ]
  818: 
  819: =item * B<required_group>
  820: 
  821: At least one parameter from the group must be specified.
  822: 
  823:   relationships => [
  824:     {
  825:       type => 'required_group',
  826:       params => ['id', 'name'],
  827:       logic => 'or',
  828:       description => 'Must specify either id or name'
  829:     }
  830:   ]
  831: 
  832: =item * B<conditional_requirement>
  833: 
  834: If one parameter is specified, another becomes required.
  835: 
  836:   relationships => [
  837:     {
  838:       type => 'conditional_requirement',
  839:       if => 'async',
  840:       then_required => 'callback',
  841:       description => 'When async is specified, callback is required'
  842:     }
  843:   ]
  844: 
  845: =item * B<dependency>
  846: 
  847: One parameter requires another to be present.
  848: 
  849:   relationships => [
  850:     {
  851:       type => 'dependency',
  852:       param => 'port',
  853:       requires => 'host',
  854:       description => 'port requires host to be specified'
  855:     }
  856:   ]
  857: 
  858: =item * B<value_constraint>
  859: 
  860: Specific value requirements between parameters.
  861: 
  862:   relationships => [
  863:     {
  864:       type => 'value_constraint',
  865:       if => 'ssl',
  866:       then => 'port',
  867:       operator => '==',
  868:       value => 443,
  869:       description => 'When ssl is specified, port must equal 443'
  870:     }
  871:   ]
  872: 
  873: =item * B<value_conditional>
  874: 
  875: Parameter required when another has a specific value.
  876: 
  877:   relationships => [
  878:     {
  879:       type => 'value_conditional',
  880:       if => 'mode',
  881:       equals => 'secure',
  882:       then_required => 'key',
  883:       description => "When mode equals 'secure', key is required"
  884:     }
  885:   ]
  886: 
  887: =back
  888: 
  889: If a parameter is optional and its value is C<undef>,
  890: validation will be skipped for that parameter.
  891: 
  892: If the validation fails, the function will C<croak> with an error message describing the validation failure.
  893: 
  894: If the validation is successful, the function will return a reference to a new hash containing the validated and (where applicable) coerced parameters.  Integer and number parameters will be coerced to their respective types.
  895: 
  896: The C<description> field is optional but recommended for clearer error messages.
  897: 
  898: =back
