   84: =head2 validate_strict
   85: 
   86: Validates a set of parameters against a schema.
   87: 
   88: This function takes two mandatory arguments:
   89: 
   90: =over 4
   91: 
   92: =item * C<schema> || C<members>
   93: 
   94: A reference to a hash that defines the validation rules for each parameter.
   95: The keys of the hash are the parameter names, and the values are either a string representing the parameter type or a reference to a hash containing more detailed rules.
   96: 
   97: As an alternative the schema may be supplied as an B<arrayref of parameter
   98: hashrefs>, where every element describes one parameter and carries a mandatory
   99: C<name> key:
  100: 
  101:   $schema = [
  102:     { name => 'username', type => 'string', min => 3, max => 50 },
  103:     { name => 'age',      type => 'integer', min => 0, max => 150 },
  104:     { name => 'role',     type => 'string', optional => 1, default => 'user' },
  105:   ];
  106: 
  107: The arrayref form is normalised to the standard hashref form before any further
  108: processing.  It is particularly useful when declaration order matters (e.g.
  109: for positional or mixed calling conventions used by some CPAN modules).  The
  110: C<name> key is consumed during normalisation and does not appear as a
  111: validation rule.
  112: 
  113: For some sort of compatibility with L<Data::Processor>,
  114: it is possible to wrap the schema within a hash like this:
  115: 
  116:   $schema = {
  117:     description => 'Describe what this schema does',
  118:     error_msg => 'An error message',
  119:     schema => {
  120:       # ... schema goes here
  121:     }
  122:   }
  123: 
  124: =item * C<args> || C<input>
  125: 
  126: A reference to a hash containing the parameters to be validated.
  127: The keys of the hash are the parameter names, and the values are the parameter values.
  128: 
  129: =back
  130: 
  131: It takes optional arguments:
  132: 
  133: =over 4
  134: 
  135: =item * C<description>
  136: 
  137: What the schema does,
  138: used in error messages.
  139: 
  140: =item * C<error_msg>
  141: 
  142: Overrides the default message when something doesn't validate.
  143: 
  144: =item * C<unknown_parameter_handler>
  145: 
  146: This parameter describes what to do when a parameter is given that is not in the schema of valid parameters.
  147: It must be one of C<die>, C<warn>, or C<ignore>.
  148: 
  149: It defaults to C<die> unless C<carp_on_warn> is given, in which case it defaults to C<warn>.
  150: 
  151: =item * C<logger>
  152: 
  153: A logging object that understands messages such as C<error> and C<warn>.
  154: 
  155: =item * C<custom_types>
  156: 
  157: A reference to a hash that defines reusable custom types.
  158: Custom types allow you to define validation rules once and reuse them throughout your schema,
  159: making your validation logic more maintainable and readable.
  160: 
  161: Each custom type is defined as a hash reference containing the same validation rules available for regular parameters
  162: (C<type>, C<min>, C<max>, C<matches>, C<memberof>, C<values>, C<enum>, C<notmemberof>, C<callback>, etc.).
  163: 
  164:   my $custom_types = {
  165:     email => {
  166:       type => 'string',
  167:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/,
  168:       error_msg => 'Invalid email address format'
  169:     }, phone => {
  170:       type => 'string',
  171:       matches => qr/^\+?[1-9]\d{1,14}$/,
  172:       min => 10,
  173:       max => 15
  174:     }, percentage => {
  175:       type => 'number',
  176:       min => 0,
  177:       max => 100
  178:     }, status => {
  179:       type => 'string',
  180:       memberof => ['draft', 'published', 'archived']
  181:     }
  182:   };
  183: 
  184:   my $schema = {
  185:     user_email => { type => 'email' },
  186:     contact_number => { type => 'phone', optional => 1 },
  187:     completion => { type => 'percentage' },
  188:     post_status => { type => 'status' }
  189:   };
  190: 
  191:   my $validated = validate_strict(
  192:     schema => $schema,
  193:     input => $input,
  194:     custom_types => $custom_types
  195:   );
  196: 
  197: Custom types can be extended or overridden in the schema by specifying additional constraints:
  198: 
  199:   my $schema = {
  200:     admin_username => {
  201:       type => 'username',  # Uses custom type definition
  202:       min => 5,            # Overrides custom type's min value
  203:       max => 15            # Overrides custom type's max value
  204:     }
  205:   };
  206: 
  207: Custom types work seamlessly with nested schema, optional parameters, and all other validation features.
  208: 
  209: =back
  210: 
  211: The schema can define the following rules for each parameter:
  212: 
  213: =over 4
  214: 
  215: =item * C<type>
  216: 
  217: The data type of the parameter.
  218: Valid types are C<string>, C<integer>, C<number>, C<float> C<boolean>, C<scalar>, C<scalarref>, C<hashref>, C<arrayref>, C<object> and C<coderef>.
  219: C<scalar> accepts any plain scalar value (string, number, boolean, etc.) but rejects references (arrayrefs, hashrefs, coderefs, objects).
  220: C<scalarref> accepts a reference to a scalar value (e.g. C<\$var>) but rejects plain scalars, arrayrefs, hashrefs, coderefs, and objects.
  221: 
  222: A type can be an arrayref when a parameter could have different types (e.g. a string or an object).
  223: 
  224:   $schema = {
  225:     username => [
  226:       { type => 'string', min => 3, max => 50 },	# Name
  227:       { type => 'integer', 'min' => 1 },	# UID that isn't root
  228:     ]
  229:   };
  230: 
  231: As a shorthand, C<type> itself may be an arrayref of type name strings (a I<union type>)
  232: when all other constraints are shared between the alternatives:
  233: 
  234:   $schema = {
  235:     data => { type => ['string', 'arrayref'] },
  236:     id   => { type => ['string', 'integer'], optional => 1 },
  237:   };
  238: 
  239: This is equivalent to the full array-of-rules form but more concise.
  240: Every other key in the rule hash (C<optional>, C<min>, C<max>, C<matches>, etc.)
  241: is inherited by each candidate type and validated independently against it.
  242: Type names are tried left-to-right; the first match wins and its coercion
  243: (e.g. numeric types) is propagated back to the caller.
  244: If the value fails all candidate types, validation croaks with a message
  245: listing the union members.
  246: 
  247: =item * C<can>
  248: 
  249: The parameter must be an object that understands the method C<can>.
  250: C<can> can be a simple scalar string of a method name,
  251: or an arrayref of a list of method names, all of which must be supported by the object.
  252: 
  253:    $schema = {
  254:      gedcom => { type => object, can => 'get_individual' }
  255:    }
  256: 
  257: =item * C<isa>
  258: 
  259: The parameter must be an object of type C<isa>.
  260: 
  261: =item * C<memberof>
  262: 
  263: The parameter must be a member of the given arrayref.
  264: 
  265:   status => {
  266:     type => 'string',
  267:     memberof => ['draft', 'published', 'archived']
  268:   }
  269: 
  270:   priority => {
  271:     type => 'integer',
  272:     memberof => [1, 2, 3, 4, 5]
  273:   }
  274: 
  275: For string types, the comparison is case-sensitive by default. Use the C<case_sensitive>
  276: flag to control this behavior:
  277: 
  278:   # Case-sensitive (default) - must be exact match
  279:   code => {
  280:     type => 'string',
  281:     memberof => ['ABC', 'DEF', 'GHI']
  282:     # 'abc' will fail
  283:   }
  284: 
  285:   # Case-insensitive - any case accepted
  286:   code => {
  287:     type => 'string',
  288:     memberof => ['ABC', 'DEF', 'GHI'],
  289:     case_sensitive => 0
  290:     # 'abc', 'Abc', 'ABC' all pass, original case preserved
  291:   }
  292: 
  293: For numeric types (C<integer>, C<number>, C<float>), the comparison uses numeric
  294: equality (C<==> operator):
  295: 
  296:   rating => {
  297:     type => 'number',
  298:     memberof => [0.5, 1.0, 1.5, 2.0]
  299:   }
  300: 
  301: Note that C<memberof> cannot be combined with C<min> or C<max> constraints as they
  302: serve conflicting purposes - C<memberof> defines an explicit whitelist while C<min>/C<max>
  303: define ranges.
  304: 
  305: =item * C<enum>
  306: 
  307: Same as C<memberof>.
  308: 
  309: =item * C<values>
  310: 
  311: Same as C<memberof>.
  312: 
  313: =item * C<notmemberof>
  314: 
  315: The parameter must not be a member of the given arrayref (blacklist).
  316: This is the inverse of C<memberof>.
  317: 
  318:   username => {
  319:     type => 'string',
  320:     notmemberof => ['admin', 'root', 'system', 'administrator']
  321:   }
  322: 
  323:   port => {
  324:     type => 'integer',
  325:     notmemberof => [22, 23, 25, 80, 443]  # Reserved ports
  326:   }
  327: 
  328: Like C<memberof>, string comparisons are case-sensitive by default but can be controlled
  329: with the C<case_sensitive> flag:
  330: 
  331:   # Case-sensitive (default)
  332:   username => {
  333:     type => 'string',
  334:     notmemberof => ['Admin', 'Root']
  335:     # 'admin' would pass, 'Admin' would fail
  336:   }
  337: 
  338:   # Case-insensitive
  339:   username => {
  340:     type => 'string',
  341:     notmemberof => ['Admin', 'Root'],
  342:     case_sensitive => 0
  343:     # 'admin', 'ADMIN', 'Admin' all fail
  344:   }
  345: 
  346: The blacklist is checked after any C<transform> rules are applied, allowing you to
  347: normalize input before checking:
  348: 
  349:   username => {
  350:     type => 'string',
  351:     transform => sub { lc($_[0]) },  # Normalize to lowercase
  352:     notmemberof => ['admin', 'root', 'system']
  353:   }
  354: 
  355: C<notmemberof> can be combined with other validation rules:
  356: 
  357:   username => {
  358:     type => 'string',
  359:     notmemberof => ['admin', 'root', 'system'],
  360:     min => 3,
  361:     max => 20,
  362:     matches => qr/^[a-z0-9_]+$/
  363:   }
  364: 
  365: =item * C<case_sensitive>
  366: 
  367: A boolean value indicating whether string comparisons should be case-sensitive.
  368: This flag affects the C<memberof> and C<notmemberof> validation rules.
  369: The default value is C<1> (case-sensitive).
  370: 
  371: When set to C<0>, string comparisons are performed case-insensitively, allowing values
  372: with different casing to match. The original case of the input value is preserved in
  373: the validated output.
  374: 
  375:   # Case-sensitive (default)
  376:   status => {
  377:     type => 'string',
  378:     memberof => ['Draft', 'Published', 'Archived'] # Input 'draft' will fail - must match exact case
  379:   }
  380: 
  381:   # Case-insensitive
  382:   status => {
  383:     type => 'string',
  384:     memberof => ['Draft', 'Published', 'Archived'],
  385:     case_sensitive => 0 # Input 'draft', 'DRAFT', or 'DrAfT' will all pass
  386:   }
  387: 
  388:   country_code => {
  389:     type => 'string',
  390:     memberof => ['US', 'UK', 'CA', 'FR'],
  391:     case_sensitive => 0  # Accept 'us', 'US', 'Us', etc.
  392:   }
  393: 
  394: This flag has no effect on numeric types (C<integer>, C<number>, C<float>) as numbers
  395: do not have case.
  396: 
  397: =item * C<min>/C<minimum>
  398: 
  399: The minimum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  400: 
  401: =item * C<max>
  402: 
  403: The maximum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  404: 
  405: =item * C<matches>
  406: 
  407: A regular expression that the parameter value must match.
  408: Checks all members of arrayrefs.
  409: 
  410: =item * C<nomatch>
  411: 
  412: A regular expression that the parameter value must not match.
  413: Checks all members of arrayrefs.
  414: 
  415: =item * C<position>
  416: 
  417: For routines and methods that take positional args,
  418: this integer value defines which position the argument will be in.
  419: If this is set for all arguments,
  420: C<validate_strict> will return a reference to an array, rather than a reference to a hash.
  421: 
  422: =item * C<regex>
  423: 
  424: Synonym of matches
  425: 
  426: =item * C<description>
  427: 
  428: The description of the rule
  429: 
  430: =item * C<callback>
  431: 
  432: A code reference to a subroutine that performs custom validation logic.
  433: The subroutine should accept the parameter value, the argument list and the schema as arguments and return true if the value is valid, false otherwise.
  434: 
  435: Use this to test more complex examples:
  436: 
  437:   my $schema = {
  438:     even_number => {
  439:       type => 'integer',
  440:       callback => sub { $_[0] % 2 == 0 }
  441:   };
  442: 
  443:   # Specify the arguments for a routine which has a second, optional argument, which, if given, must be less than or equal to the first
  444:   my $schema = {
  445:     first => {
  446:       type => 'integer'
  447:     }, second => {
  448:       type => 'integer',
  449:       optional => 1,
  450:       callback => sub {
  451:         my($value, $args) = @_;
  452: 	# The 'defined' is needed in case 'second' is evaluated before 'first'
  453: 	return (defined($args->{first}) && $value <= $args->{first}) ? 1 : 0
  454:       }
  455:     }
  456:   };
  457: 
  458: =item * C<optional>
  459: 
  460: A boolean value indicating whether the parameter is optional.
  461: If true, the parameter is not required.
  462: If false or omitted, the parameter is required.
  463: 
  464: It can be a reference to a code snippet that will return true or false,
  465: to determine if the parameter is optional or not.
  466: The code will be called with two arguments: the value of the parameter and hash ref of all parameters:
  467: 
  468:   my $schema = {
  469:     optional_field => {
  470:       type => 'string',
  471:       optional => sub {
  472:         my ($value, $all_params) = @_;
  473:         return $all_params->{make_optional} ? 1 : 0;
  474:       }
  475:     },
  476:     make_optional => { type => 'boolean' }
  477:   };
  478: 
  479:   my $result = validate_strict(schema => $schema, input => { make_optional => 1 });
  480: 
  481: If the parameter is not optional, it can be passed an undef value, which will not flag an error.
  482: This is by design.
  483: So this will not say that the required parameter 's' is missing:
  484: 
  485:     validate_strict(
  486:         schema => { s => { type => 'string' } },
  487:         input  => { s => undef },
  488:     );
  489: 
  490: =item * C<default>
  491: 
  492: Populate missing optional parameters with the specified value.
  493: Note that this value is not validated.
  494: 
  495:   username => {
  496:     type => 'string',
  497:     optional => 1,
  498:     default => 'guest'
  499:   }
  500: 
  501: =item * C<element_type>
  502: 
  503: Extends the validation to individual elements of arrays.
  504: 
  505:   tags => {
  506:     type => 'arrayref',
  507:     element_type => 'number',	# Float means the same
  508:     min => 1,	# this is the length of the array, not the min value for each of the numbers. For that, add a C<schema> rule
  509:     max => 5
  510:   }
  511: 
  512: =item * C<error_msg>
  513: 
  514: The custom error message to be used in the event of a validation failure.
  515: 
  516:   age => {
  517:     type => 'integer',
  518:     min => 18,
  519:     error_msg => 'You must be at least 18 years old'
  520:   }
  521: 
  522: =item * C<nullable>
  523: 
  524: Like optional,
  525: though this cannot be a coderef,
  526: only a flag.
  527: 
  528: =item * C<schema>
  529: 
  530: You can validate nested hashrefs and arrayrefs using the C<schema> property:
  531: 
  532:     my $schema = {
  533:         user => {	# 'user' is a hashref
  534:             type => 'hashref',
  535:             schema => {	# Specify what the elements of the hash should be
  536:                 name => { type => 'string' },
  537:                 age => { type => 'integer', min => 0 },
  538:                 hobbies => {	# 'hobbies' is an array ref that this user has
  539:                     type => 'arrayref',
  540:                     schema => { type => 'string' }, # Validate each hobby
  541:                     min => 1 # At least one hobby
  542:                 }
  543:             }
  544:         }, metadata => {
  545:             type => 'hashref',
  546:             schema => {
  547:                 created => { type => 'string' },
  548:                 tags => {
  549:                     type => 'arrayref',
  550:                     schema => {
  551:                         type => 'string',
  552:                         matches => qr/^[a-z]+$/	# Or you can say matches => '^[a-z]+$'
  553:                     }
  554:                 }
  555:             }
  556:         }
  557:     };
  558: 
  559: =item * C<validate>
  560: 
  561: A snippet of code that validates the input.
  562: It's passed the input arguments,
  563: and return a string containing a reason for rejection,
  564: or undef if it's allowed.
  565: 
  566:     my $schema = {
  567:       user => {
  568:         type => 'string',
  569: 	validate => sub {
  570: 	  if($_[0]->{'password'} eq 'bar') {
  571: 	    return undef;
  572: 	  }
  573: 	  return 'Invalid password, try again';
  574: 	}
  575:       }, password => {
  576:          type => 'string'
  577:       }
  578:     };
  579: 
  580: =item * C<transform>
  581: 
  582: A code reference to a subroutine that transforms/sanitizes the parameter value before validation.
  583: The subroutine should accept the parameter value as an argument and return the transformed value.
  584: The transformation is applied before any validation rules are checked, allowing you to normalize
  585: or clean data before it is validated.
  586: 
  587: Common use cases include trimming whitespace, normalizing case, formatting phone numbers,
  588: sanitizing user input, and converting between data formats.
  589: 
  590:   # Simple string transformations
  591:   username => {
  592:     type => 'string',
  593:     transform => sub { lc(trim($_[0])) },  # lowercase and trim
  594:     matches => qr/^[a-z0-9_]+$/
  595:   }
  596: 
  597:   email => {
  598:     type => 'string',
  599:     transform => sub { lc(trim($_[0])) },  # normalize email
  600:     matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  601:   }
  602: 
  603:   # Array transformations
  604:   tags => {
  605:     type => 'arrayref',
  606:     transform => sub { [map { lc($_) } @{$_[0]}] },  # lowercase all elements
  607:     element_type => 'string'
  608:   }
  609: 
  610:   keywords => {
  611:     type => 'arrayref',
  612:     transform => sub {
  613:       my @arr = map { lc(trim($_)) } @{$_[0]};
  614:       my %seen;
  615:       return [grep { !$seen{$_}++ } @arr];  # remove duplicates
  616:     }
  617:   }
  618: 
  619:   # Numeric transformations
  620:   quantity => {
  621:     type => 'integer',
  622:     transform => sub { int($_[0] + 0.5) },  # round to nearest integer
  623:     min => 1
  624:   }
  625: 
  626:   # Sanitization
  627:   slug => {
  628:     type => 'string',
  629:     transform => sub {
  630:       my $str = lc(trim($_[0]));
  631:       $str =~ s/[^\w\s-]//g;  # remove special characters
  632:       $str =~ s/\s+/-/g;      # replace spaces with hyphens
  633:       return $str;
  634:     },
  635:     matches => qr/^[a-z0-9-]+$/
  636:   }
  637: 
  638:   phone => {
  639:     type => 'string',
  640:     transform => sub {
  641:       my $str = $_[0];
  642:       $str =~ s/\D//g;  # remove all non-digits

  678: handles edge cases appropriately.

  680: since that is what will be validated.

  698: =item * C<cross_validation>
  699: 
  700: A reference to a hash that defines validation rules that depend on more than one parameter.
  701: Cross-field validations are performed after all individual parameter validations have passed,
  702: allowing you to enforce business logic that requires checking relationships between different fields.
  703: 
  704: Each cross-validation rule is a key-value pair where the key is a descriptive name for the validation
  705: and the value is a code reference that accepts a hash reference of all validated parameters.
  706: The subroutine should return C<undef> if the validation passes, or an error message string if it fails.
  707: 
  708:   my $schema = {
  709:     password => { type => 'string', min => 8 },
  710:     password_confirm => { type => 'string' }
  711:   };
  712: 
  713:   my $cross_validation = {
  714:     passwords_match => sub {
  715:       my $params = shift;
  716:       return $params->{password} eq $params->{password_confirm}
  717:         ? undef : "Passwords don't match";
  718:     }
  719:   };
  720: 
  721:   my $validated = validate_strict(
  722:     schema => $schema,
  723:     input => $input,
  724:     cross_validation => $cross_validation
  725:   );
  726: 
  727: Common use cases include password confirmation, date range validation, numeric comparisons,
  728: and conditional requirements:
  729: 
  730:   # Date range validation
  731:   my $cross_validation = {
  732:     date_range_valid => sub {
  733:       my $params = shift;
  734:       return $params->{start_date} le $params->{end_date}
  735:         ? undef : "Start date must be before or equal to end date";
  736:     }
  737:   };
  738: 
  739:   # Price range validation
  740:   my $cross_validation = {
  741:     price_range_valid => sub {
  742:       my $params = shift;
  743:       return $params->{min_price} <= $params->{max_price}
  744:         ? undef : "Minimum price must be less than or equal to maximum price";
  745:     }
  746:   };
  747: 
  748:   # Conditional required field
  749:   my $cross_validation = {
  750:     address_required_for_delivery => sub {
  751:       my $params = shift;
  752:       if ($params->{shipping_method} eq 'delivery' && !$params->{delivery_address}) {
  753:         return "Delivery address is required when shipping method is 'delivery'";
  754:       }
  755:       return undef;
  756:     }
  757:   };
  758: 
  759: Multiple cross-validations can be defined in the same hash, and they are all checked in order.
  760: If any cross-validation fails, the function will C<croak> with the error message returned by the validation:
  761: 
  762:   my $cross_validation = {
  763:     passwords_match => sub {
  764:       my $params = shift;
  765:       return $params->{password} eq $params->{password_confirm}
  766:         ? undef : "Passwords don't match";
  767:     },

  770:       return $params->{email} eq $params->{email_confirm}
  771:         ? undef : "Email addresses don't match";

  778:         ? undef : "Age doesn't match birth year";
  779:     }
  780:   };

  782: Cross-validations receive the parameters after individual validation and transformation have been applied,
  783: so you can rely on the data being in the correct format and type:
  784: 
  785:   my $schema = {

  828: =item * C<relationships>
  829: 
  830: A reference to an array that defines validation rules based on relationships between parameters.
  831: Relationship validations are performed after all individual parameter validations have passed,
  832: but before cross-validations.
  833: 
  834: Each relationship is a hash reference with a C<type> field and additional fields depending on the type:
  835: 
  836: =over 4
  837: 
  838: =item * B<mutually_exclusive>
  839: 
  840: Parameters that cannot be specified together.
  841: 
  842:   relationships => [
  843:     {
  844:       type => 'mutually_exclusive',
  845:       params => ['file', 'content'],
  846:       description => 'Cannot specify both file and content'
  847:     }
  848:   ]
  849: 
  850: =item * B<required_group>
  851: 
  852: At least one parameter from the group must be specified.
  853: 
  854:   relationships => [
  855:     {
  856:       type => 'required_group',
  857:       params => ['id', 'name'],
  858:       logic => 'or',
  859:       description => 'Must specify either id or name'
  860:     }
  861:   ]
  862: 
  863: =item * B<conditional_requirement>
  864: 
  865: If one parameter is specified, another becomes required.
  866: 
  867:   relationships => [
  868:     {
  869:       type => 'conditional_requirement',
  870:       if => 'async',
  871:       then_required => 'callback',
  872:       description => 'When async is specified, callback is required'
  873:     }
  874:   ]
  875: 
  876: =item * B<dependency>
  877: 
  878: One parameter requires another to be present.
  879: 
  880:   relationships => [
  881:     {
  882:       type => 'dependency',
  883:       param => 'port',
  884:       requires => 'host',
  885:       description => 'port requires host to be specified'
  886:     }
  887:   ]
  888: 
  889: =item * B<value_constraint>
  890: 
  891: Specific value requirements between parameters.
  892: 
  893:   relationships => [
  894:     {
  895:       type => 'value_constraint',
  896:       if => 'ssl',
  897:       then => 'port',
  898:       operator => '==',
  899:       value => 443,
  900:       description => 'When ssl is specified, port must equal 443'
  901:     }
  902:   ]
  903: 
  904: =item * B<value_conditional>
  905: 
  906: Parameter required when another has a specific value.
  907: 
  908:   relationships => [
  909:     {
  910:       type => 'value_conditional',
  911:       if => 'mode',
  912:       equals => 'secure',
  913:       then_required => 'key',
  914:       description => "When mode equals 'secure', key is required"
  915:     }
  916:   ]
  917: 
  918: =back
  919: 
  920: If a parameter is optional and its value is C<undef>,
  921: validation will be skipped for that parameter.
  922: 
  923: If the validation fails, the function will C<croak> with an error message describing the validation failure.
  924: 
  925: If the validation is successful, the function will return a reference to a new hash containing the validated and (where applicable) coerced parameters.  Integer and number parameters will be coerced to their respective types.
  926: 
  927: The C<description> field is optional but recommended for clearer error messages.
  928: 
  929: =back
