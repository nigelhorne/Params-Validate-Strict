   83: =head2 validate_strict
   84: 
   85: Validates a set of parameters against a schema.
   86: 
   87: This function takes two mandatory arguments:
   88: 
   89: =over 4
   90: 
   91: =item * C<schema> || C<members>
   92: 
   93: A reference to a hash that defines the validation rules for each parameter.
   94: The keys of the hash are the parameter names, and the values are either a string representing the parameter type or a reference to a hash containing more detailed rules.
   95: 
   96: As an alternative the schema may be supplied as an B<arrayref of parameter
   97: hashrefs>, where every element describes one parameter and carries a mandatory
   98: C<name> key:
   99: 
  100:   $schema = [
  101:     { name => 'username', type => 'string', min => 3, max => 50 },
  102:     { name => 'age',      type => 'integer', min => 0, max => 150 },
  103:     { name => 'role',     type => 'string', optional => 1, default => 'user' },
  104:   ];
  105: 
  106: The arrayref form is normalised to the standard hashref form before any further
  107: processing.  It is particularly useful when declaration order matters (e.g.
  108: for positional or mixed calling conventions used by some CPAN modules).  The
  109: C<name> key is consumed during normalisation and does not appear as a
  110: validation rule.
  111: 
  112: For some sort of compatibility with L<Data::Processor>,
  113: it is possible to wrap the schema within a hash like this:
  114: 
  115:   $schema = {
  116:     description => 'Describe what this schema does',
  117:     error_msg => 'An error message',
  118:     schema => {
  119:       # ... schema goes here
  120:     }
  121:   }
  122: 
  123: =item * C<args> || C<input>
  124: 
  125: A reference to a hash containing the parameters to be validated.
  126: The keys of the hash are the parameter names, and the values are the parameter values.
  127: 
  128: =back
  129: 
  130: It takes optional arguments:
  131: 
  132: =over 4
  133: 
  134: =item * C<description>
  135: 
  136: What the schema does,
  137: used in error messages.
  138: 
  139: =item * C<error_msg>
  140: 
  141: Overrides the default message when something doesn't validate.
  142: 
  143: =item * C<unknown_parameter_handler>
  144: 
  145: This parameter describes what to do when a parameter is given that is not in the schema of valid parameters.
  146: It must be one of C<die>, C<warn>, or C<ignore>.
  147: 
  148: It defaults to C<die> unless C<carp_on_warn> is given, in which case it defaults to C<warn>.
  149: 
  150: =item * C<logger>
  151: 
  152: A logging object that understands messages such as C<error> and C<warn>.
  153: 
  154: =item * C<custom_types>
  155: 
  156: A reference to a hash that defines reusable custom types.
  157: Custom types allow you to define validation rules once and reuse them throughout your schema,
  158: making your validation logic more maintainable and readable.
  159: 
  160: Each custom type is defined as a hash reference containing the same validation rules available for regular parameters
  161: (C<type>, C<min>, C<max>, C<matches>, C<memberof>, C<enum>, C<notmemberof>, C<callback>, etc.).
  162: 
  163:   my $custom_types = {
  164:     email => {
  165:       type => 'string',
  166:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/,
  167:       error_msg => 'Invalid email address format'
  168:     }, phone => {
  169:       type => 'string',
  170:       matches => qr/^\+?[1-9]\d{1,14}$/,
  171:       min => 10,
  172:       max => 15
  173:     }, percentage => {
  174:       type => 'number',
  175:       min => 0,
  176:       max => 100
  177:     }, status => {
  178:       type => 'string',
  179:       memberof => ['draft', 'published', 'archived']
  180:     }
  181:   };
  182: 
  183:   my $schema = {
  184:     user_email => { type => 'email' },
  185:     contact_number => { type => 'phone', optional => 1 },
  186:     completion => { type => 'percentage' },
  187:     post_status => { type => 'status' }
  188:   };
  189: 
  190:   my $validated = validate_strict(
  191:     schema => $schema,
  192:     input => $input,
  193:     custom_types => $custom_types
  194:   );
  195: 
  196: Custom types can be extended or overridden in the schema by specifying additional constraints:
  197: 
  198:   my $schema = {
  199:     admin_username => {
  200:       type => 'username',  # Uses custom type definition
  201:       min => 5,            # Overrides custom type's min value
  202:       max => 15            # Overrides custom type's max value
  203:     }
  204:   };
  205: 
  206: Custom types work seamlessly with nested schema, optional parameters, and all other validation features.
  207: 
  208: =back
  209: 
  210: The schema can define the following rules for each parameter:
  211: 
  212: =over 4
  213: 
  214: =item * C<type>
  215: 
  216: The data type of the parameter.
  217: Valid types are C<string>, C<integer>, C<number>, C<float> C<boolean>, C<hashref>, C<arrayref>, C<object> and C<coderef>.
  218: 
  219: A type can be an arrayref when a parameter could have different types (e.g. a string or an object).
  220: 
  221:   $schema = {
  222:     username => [
  223:       { type => 'string', min => 3, max => 50 },	# Name
  224:       { type => 'integer', 'min' => 1 },	# UID that isn't root
  225:     ]
  226:   };
  227: 
  228: As a shorthand, C<type> itself may be an arrayref of type name strings (a I<union type>)
  229: when all other constraints are shared between the alternatives:
  230: 
  231:   $schema = {
  232:     data => { type => ['string', 'arrayref'] },
  233:     id   => { type => ['string', 'integer'], optional => 1 },
  234:   };
  235: 
  236: This is equivalent to the full array-of-rules form but more concise.
  237: Every other key in the rule hash (C<optional>, C<min>, C<max>, C<matches>, etc.)
  238: is inherited by each candidate type and validated independently against it.
  239: Type names are tried left-to-right; the first match wins and its coercion
  240: (e.g. numeric types) is propagated back to the caller.
  241: If the value fails all candidate types, validation croaks with a message
  242: listing the union members.
  243: 
  244: =item * C<can>
  245: 
  246: The parameter must be an object that understands the method C<can>.
  247: C<can> can be a simple scalar string of a method name,
  248: or an arrayref of a list of method names, all of which must be supported by the object.
  249: 
  250:    $schema = {
  251:      gedcom => { type => object, can => 'get_individual' }
  252:    }
  253: 
  254: =item * C<isa>
  255: 
  256: The parameter must be an object of type C<isa>.
  257: 
  258: =item * C<memberof>
  259: 
  260: The parameter must be a member of the given arrayref.
  261: 
  262:   status => {
  263:     type => 'string',
  264:     memberof => ['draft', 'published', 'archived']
  265:   }
  266: 
  267:   priority => {
  268:     type => 'integer',
  269:     memberof => [1, 2, 3, 4, 5]
  270:   }
  271: 
  272: For string types, the comparison is case-sensitive by default. Use the C<case_sensitive>
  273: flag to control this behavior:
  274: 
  275:   # Case-sensitive (default) - must be exact match
  276:   code => {
  277:     type => 'string',
  278:     memberof => ['ABC', 'DEF', 'GHI']
  279:     # 'abc' will fail
  280:   }
  281: 
  282:   # Case-insensitive - any case accepted
  283:   code => {
  284:     type => 'string',
  285:     memberof => ['ABC', 'DEF', 'GHI'],
  286:     case_sensitive => 0
  287:     # 'abc', 'Abc', 'ABC' all pass, original case preserved
  288:   }
  289: 
  290: For numeric types (C<integer>, C<number>, C<float>), the comparison uses numeric
  291: equality (C<==> operator):
  292: 
  293:   rating => {
  294:     type => 'number',
  295:     memberof => [0.5, 1.0, 1.5, 2.0]
  296:   }
  297: 
  298: Note that C<memberof> cannot be combined with C<min> or C<max> constraints as they
  299: serve conflicting purposes - C<memberof> defines an explicit whitelist while C<min>/C<max>
  300: define ranges.
  301: 
  302: =item * C<enum>
  303: 
  304: Same as C<memberof>.
  305: 
  306: =item * C<notmemberof>
  307: 
  308: The parameter must not be a member of the given arrayref (blacklist).
  309: This is the inverse of C<memberof>.
  310: 
  311:   username => {
  312:     type => 'string',
  313:     notmemberof => ['admin', 'root', 'system', 'administrator']
  314:   }
  315: 
  316:   port => {
  317:     type => 'integer',
  318:     notmemberof => [22, 23, 25, 80, 443]  # Reserved ports
  319:   }
  320: 
  321: Like C<memberof>, string comparisons are case-sensitive by default but can be controlled
  322: with the C<case_sensitive> flag:
  323: 
  324:   # Case-sensitive (default)
  325:   username => {
  326:     type => 'string',
  327:     notmemberof => ['Admin', 'Root']
  328:     # 'admin' would pass, 'Admin' would fail
  329:   }
  330: 
  331:   # Case-insensitive
  332:   username => {
  333:     type => 'string',
  334:     notmemberof => ['Admin', 'Root'],
  335:     case_sensitive => 0
  336:     # 'admin', 'ADMIN', 'Admin' all fail
  337:   }
  338: 
  339: The blacklist is checked after any C<transform> rules are applied, allowing you to
  340: normalize input before checking:
  341: 
  342:   username => {
  343:     type => 'string',
  344:     transform => sub { lc($_[0]) },  # Normalize to lowercase
  345:     notmemberof => ['admin', 'root', 'system']
  346:   }
  347: 
  348: C<notmemberof> can be combined with other validation rules:
  349: 
  350:   username => {
  351:     type => 'string',
  352:     notmemberof => ['admin', 'root', 'system'],
  353:     min => 3,
  354:     max => 20,
  355:     matches => qr/^[a-z0-9_]+$/
  356:   }
  357: 
  358: =item * C<case_sensitive>
  359: 
  360: A boolean value indicating whether string comparisons should be case-sensitive.
  361: This flag affects the C<memberof> and C<notmemberof> validation rules.
  362: The default value is C<1> (case-sensitive).
  363: 
  364: When set to C<0>, string comparisons are performed case-insensitively, allowing values
  365: with different casing to match. The original case of the input value is preserved in
  366: the validated output.
  367: 
  368:   # Case-sensitive (default)
  369:   status => {
  370:     type => 'string',
  371:     memberof => ['Draft', 'Published', 'Archived'] # Input 'draft' will fail - must match exact case
  372:   }
  373: 
  374:   # Case-insensitive
  375:   status => {
  376:     type => 'string',
  377:     memberof => ['Draft', 'Published', 'Archived'],
  378:     case_sensitive => 0 # Input 'draft', 'DRAFT', or 'DrAfT' will all pass
  379:   }
  380: 
  381:   country_code => {
  382:     type => 'string',
  383:     memberof => ['US', 'UK', 'CA', 'FR'],
  384:     case_sensitive => 0  # Accept 'us', 'US', 'Us', etc.
  385:   }
  386: 
  387: This flag has no effect on numeric types (C<integer>, C<number>, C<float>) as numbers
  388: do not have case.
  389: 
  390: =item * C<min>/C<minimum>
  391: 
  392: The minimum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  393: 
  394: =item * C<max>
  395: 
  396: The maximum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  397: 
  398: =item * C<matches>
  399: 
  400: A regular expression that the parameter value must match.
  401: Checks all members of arrayrefs.
  402: 
  403: =item * C<nomatch>
  404: 
  405: A regular expression that the parameter value must not match.
  406: Checks all members of arrayrefs.
  407: 
  408: =item * C<position>
  409: 
  410: For routines and methods that take positional args,
  411: this integer value defines which position the argument will be in.
  412: If this is set for all arguments,
  413: C<validate_strict> will return a reference to an array, rather than a reference to a hash.
  414: 
  415: =item * C<regex>
  416: 
  417: Synonym of matches
  418: 
  419: =item * C<description>
  420: 
  421: The description of the rule
  422: 
  423: =item * C<callback>
  424: 
  425: A code reference to a subroutine that performs custom validation logic.
  426: The subroutine should accept the parameter value, the argument list and the schema as arguments and return true if the value is valid, false otherwise.
  427: 
  428: Use this to test more complex examples:
  429: 
  430:   my $schema = {
  431:     even_number => {
  432:       type => 'integer',
  433:       callback => sub { $_[0] % 2 == 0 }
  434:   };
  435: 
  436:   # Specify the arguments for a routine which has a second, optional argument, which, if given, must be less than or equal to the first
  437:   my $schema = {
  438:     first => {
  439:       type => 'integer'
  440:     }, second => {
  441:       type => 'integer',
  442:       optional => 1,
  443:       callback => sub {
  444:         my($value, $args) = @_;
  445: 	# The 'defined' is needed in case 'second' is evaluated before 'first'
  446: 	return (defined($args->{first}) && $value <= $args->{first}) ? 1 : 0
  447:       }
  448:     }
  449:   };
  450: 
  451: =item * C<optional>
  452: 
  453: A boolean value indicating whether the parameter is optional.
  454: If true, the parameter is not required.
  455: If false or omitted, the parameter is required.
  456: 
  457: It can be a reference to a code snippet that will return true or false,
  458: to determine if the parameter is optional or not.
  459: The code will be called with two arguments: the value of the parameter and hash ref of all parameters:
  460: 
  461:   my $schema = {
  462:     optional_field => {
  463:       type => 'string',
  464:       optional => sub {
  465:         my ($value, $all_params) = @_;
  466:         return $all_params->{make_optional} ? 1 : 0;
  467:       }
  468:     },
  469:     make_optional => { type => 'boolean' }
  470:   };
  471: 
  472:   my $result = validate_strict(schema => $schema, input => { make_optional => 1 });
  473: 
  474: =item * C<default>
  475: 
  476: Populate missing optional parameters with the specified value.
  477: Note that this value is not validated.
  478: 
  479:   username => {
  480:     type => 'string',
  481:     optional => 1,
  482:     default => 'guest'
  483:   }
  484: 
  485: =item * C<element_type>
  486: 
  487: Extends the validation to individual elements of arrays.
  488: 
  489:   tags => {
  490:     type => 'arrayref',
  491:     element_type => 'number',	# Float means the same
  492:     min => 1,	# this is the length of the array, not the min value for each of the numbers. For that, add a C<schema> rule
  493:     max => 5
  494:   }

  496: =item * C<error_msg>
  497: 
  498: The custom error message to be used in the event of a validation failure.
  499: 
  500:   age => {
  501:     type => 'integer',
  502:     min => 18,
  503:     error_msg => 'You must be at least 18 years old'
  504:   }

  506: =item * C<nullable>
  507: 
  508: Like optional,
  509: though this cannot be a coderef,
  510: only a flag.

  512: =item * C<schema>
  513: 
  514: You can validate nested hashrefs and arrayrefs using the C<schema> property:
  515: 
  516:     my $schema = {
  517:         user => {	# 'user' is a hashref
  518:             type => 'hashref',
  519:             schema => {	# Specify what the elements of the hash should be
  520:                 name => { type => 'string' },
  521:                 age => { type => 'integer', min => 0 },
  522:                 hobbies => {	# 'hobbies' is an array ref that this user has
  523:                     type => 'arrayref',
  524:                     schema => { type => 'string' }, # Validate each hobby
  525:                     min => 1 # At least one hobby
  526:                 }
  527:             }
  528:         }, metadata => {
  529:             type => 'hashref',
  530:             schema => {
  531:                 created => { type => 'string' },
  532:                 tags => {
  533:                     type => 'arrayref',
  534:                     schema => {
  535:                         type => 'string',
  536:                         matches => qr/^[a-z]+$/	# Or you can say matches => '^[a-z]+$'
  537:                     }
  538:                 }
  539:             }
  540:         }
  541:     };

  543: =item * C<validate>
  544: 
  545: A snippet of code that validates the input.
  546: It's passed the input arguments,
  547: and return a string containing a reason for rejection,
  548: or undef if it's allowed.
  549: 
  550:     my $schema = {
  551:       user => {
  552:         type => 'string',
  553: 	validate => sub {
  554: 	  if($_[0]->{'password'} eq 'bar') {
  555: 	    return undef;
  556: 	  }
  557: 	  return 'Invalid password, try again';
  558: 	}
  559:       }, password => {
  560:          type => 'string'
  561:       }
  562:     };

  564: =item * C<transform>
  565: 
  566: A code reference to a subroutine that transforms/sanitizes the parameter value before validation.
  567: The subroutine should accept the parameter value as an argument and return the transformed value.
  568: The transformation is applied before any validation rules are checked, allowing you to normalize
  569: or clean data before it is validated.
  570: 
  571: Common use cases include trimming whitespace, normalizing case, formatting phone numbers,
  572: sanitizing user input, and converting between data formats.
  573: 
  574:   # Simple string transformations
  575:   username => {
  576:     type => 'string',
  577:     transform => sub { lc(trim($_[0])) },  # lowercase and trim
  578:     matches => qr/^[a-z0-9_]+$/
  579:   }
  580: 
  581:   email => {
  582:     type => 'string',
  583:     transform => sub { lc(trim($_[0])) },  # normalize email
  584:     matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  585:   }
  586: 
  587:   # Array transformations
  588:   tags => {
  589:     type => 'arrayref',
  590:     transform => sub { [map { lc($_) } @{$_[0]}] },  # lowercase all elements
  591:     element_type => 'string'
  592:   }
  593: 
  594:   keywords => {
  595:     type => 'arrayref',
  596:     transform => sub {
  597:       my @arr = map { lc(trim($_)) } @{$_[0]};
  598:       my %seen;
  599:       return [grep { !$seen{$_}++ } @arr];  # remove duplicates
  600:     }
  601:   }
  602: 
  603:   # Numeric transformations
  604:   quantity => {
  605:     type => 'integer',
  606:     transform => sub { int($_[0] + 0.5) },  # round to nearest integer
  607:     min => 1
  608:   }
  609: 
  610:   # Sanitization
  611:   slug => {
  612:     type => 'string',
  613:     transform => sub {
  614:       my $str = lc(trim($_[0]));
  615:       $str =~ s/[^\w\s-]//g;  # remove special characters
  616:       $str =~ s/\s+/-/g;      # replace spaces with hyphens
  617:       return $str;
  618:     },
  619:     matches => qr/^[a-z0-9-]+$/
  620:   }
  621: 
  622:   phone => {
  623:     type => 'string',
  624:     transform => sub {
  625:       my $str = $_[0];
  626:       $str =~ s/\D//g;  # remove all non-digits
  627:       return $str;
  628:     },
  629:     matches => qr/^\d{10}$/
  630:   }
  631: 
  632: The C<transform> function is applied to the value before any validation checks (C<min>/C<minimum>, C<max>,
  633: C<matches>, C<callback>, etc.), ensuring that validation rules are checked against the cleaned data.
  634: 
  635: Transformations work with all parameter types including nested structures:
  636: 
  637:   user => {
  638:     type => 'hashref',
  639:     schema => {
  640:       name => {
  641:         type => 'string',
  642:         transform => sub { trim($_[0]) }
  643:       },
  644:       email => {
  645:         type => 'string',
  646:         transform => sub { lc(trim($_[0])) }
  647:       }
  648:     }

  683: =item * C<cross_validation>
  684: 
  685: A reference to a hash that defines validation rules that depend on more than one parameter.
  686: Cross-field validations are performed after all individual parameter validations have passed,
  687: allowing you to enforce business logic that requires checking relationships between different fields.
  688: 
  689: Each cross-validation rule is a key-value pair where the key is a descriptive name for the validation
  690: and the value is a code reference that accepts a hash reference of all validated parameters.
  691: The subroutine should return C<undef> if the validation passes, or an error message string if it fails.
  692: 
  693:   my $schema = {
  694:     password => { type => 'string', min => 8 },
  695:     password_confirm => { type => 'string' }
  696:   };
  697: 
  698:   my $cross_validation = {
  699:     passwords_match => sub {
  700:       my $params = shift;
  701:       return $params->{password} eq $params->{password_confirm}
  702:         ? undef : "Passwords don't match";
  703:     }
  704:   };
  705: 
  706:   my $validated = validate_strict(
  707:     schema => $schema,
  708:     input => $input,
  709:     cross_validation => $cross_validation
  710:   );
  711: 
  712: Common use cases include password confirmation, date range validation, numeric comparisons,
  713: and conditional requirements:
  714: 
  715:   # Date range validation
  716:   my $cross_validation = {
  717:     date_range_valid => sub {
  718:       my $params = shift;
  719:       return $params->{start_date} le $params->{end_date}
  720:         ? undef : "Start date must be before or equal to end date";
  721:     }
  722:   };
  723: 
  724:   # Price range validation
  725:   my $cross_validation = {
  726:     price_range_valid => sub {
  727:       my $params = shift;
  728:       return $params->{min_price} <= $params->{max_price}
  729:         ? undef : "Minimum price must be less than or equal to maximum price";
  730:     }
  731:   };
  732: 
  733:   # Conditional required field
  734:   my $cross_validation = {
  735:     address_required_for_delivery => sub {
  736:       my $params = shift;
  737:       if ($params->{shipping_method} eq 'delivery' && !$params->{delivery_address}) {
  738:         return "Delivery address is required when shipping method is 'delivery'";
  739:       }
  740:       return undef;
  741:     }
  742:   };

  744: Multiple cross-validations can be defined in the same hash, and they are all checked in order.
  745: If any cross-validation fails, the function will C<croak> with the error message returned by the validation:
  746: 
  747:   my $cross_validation = {

  813: =item * C<relationships>
  814: 
  815: A reference to an array that defines validation rules based on relationships between parameters.
  816: Relationship validations are performed after all individual parameter validations have passed,
  817: but before cross-validations.
  818: 
  819: Each relationship is a hash reference with a C<type> field and additional fields depending on the type:
  820: 
  821: =over 4
  822: 
  823: =item * B<mutually_exclusive>
  824: 
  825: Parameters that cannot be specified together.
  826: 
  827:   relationships => [
  828:     {
  829:       type => 'mutually_exclusive',
  830:       params => ['file', 'content'],
  831:       description => 'Cannot specify both file and content'
  832:     }
  833:   ]
  834: 
  835: =item * B<required_group>
  836: 
  837: At least one parameter from the group must be specified.
  838: 
  839:   relationships => [
  840:     {
  841:       type => 'required_group',
  842:       params => ['id', 'name'],
  843:       logic => 'or',
  844:       description => 'Must specify either id or name'
  845:     }
  846:   ]
  847: 
  848: =item * B<conditional_requirement>
  849: 
  850: If one parameter is specified, another becomes required.
  851: 
  852:   relationships => [
  853:     {
  854:       type => 'conditional_requirement',
  855:       if => 'async',
  856:       then_required => 'callback',
  857:       description => 'When async is specified, callback is required'
  858:     }
  859:   ]
  860: 
  861: =item * B<dependency>
  862: 
  863: One parameter requires another to be present.
  864: 
  865:   relationships => [
  866:     {
  867:       type => 'dependency',
  868:       param => 'port',
  869:       requires => 'host',
  870:       description => 'port requires host to be specified'
  871:     }
  872:   ]
  873: 
  874: =item * B<value_constraint>
  875: 
  876: Specific value requirements between parameters.
  877: 
  878:   relationships => [
  879:     {
  880:       type => 'value_constraint',
  881:       if => 'ssl',
  882:       then => 'port',
  883:       operator => '==',
  884:       value => 443,
  885:       description => 'When ssl is specified, port must equal 443'
  886:     }
  887:   ]
  888: 
  889: =item * B<value_conditional>
  890: 
  891: Parameter required when another has a specific value.
  892: 
  893:   relationships => [
  894:     {
  895:       type => 'value_conditional',
  896:       if => 'mode',
  897:       equals => 'secure',
  898:       then_required => 'key',
  899:       description => "When mode equals 'secure', key is required"
  900:     }
  901:   ]
  902: 
  903: =back
  904: 
  905: If a parameter is optional and its value is C<undef>,
  906: validation will be skipped for that parameter.
  907: 
  908: If the validation fails, the function will C<croak> with an error message describing the validation failure.
  909: 
  910: If the validation is successful, the function will return a reference to a new hash containing the validated and (where applicable) coerced parameters.  Integer and number parameters will be coerced to their respective types.
  911: 
  912: The C<description> field is optional but recommended for clearer error messages.
  913: 
  914: =back
