   84: =head2 validate_strict
   85: 
   86: Validates a set of parameters against a schema.
   87: 
   88: This function takes two mandatory arguments:
   89: 
   90: =over 4
   91: 
   92: =item * C<schema> || C<members>
   93: 
   94: A reference to a hash that defines the validation rules for each parameter.
   95: The keys of the hash are the parameter names, and the values are either a string representing the parameter type or a reference to a hash containing more detailed rules.
   96: 
   97: As an alternative the schema may be supplied as an B<arrayref of parameter
   98: hashrefs>, where every element describes one parameter and carries a mandatory
   99: C<name> key:
  100: 
  101:   $schema = [
  102:     { name => 'username', type => 'string', min => 3, max => 50 },
  103:     { name => 'age',      type => 'integer', min => 0, max => 150 },
  104:     { name => 'role',     type => 'string', optional => 1, default => 'user' },
  105:   ];
  106: 
  107: The arrayref form is normalised to the standard hashref form before any further
  108: processing.  It is particularly useful when declaration order matters (e.g.
  109: for positional or mixed calling conventions used by some CPAN modules).  The
  110: C<name> key is consumed during normalisation and does not appear as a
  111: validation rule.
  112: 
  113: For some sort of compatibility with L<Data::Processor>,
  114: it is possible to wrap the schema within a hash like this:
  115: 
  116:   $schema = {
  117:     description => 'Describe what this schema does',
  118:     error_msg => 'An error message',
  119:     schema => {
  120:       # ... schema goes here
  121:     }
  122:   }
  123: 
  124: =item * C<args> || C<input>
  125: 
  126: A reference to a hash containing the parameters to be validated.
  127: The keys of the hash are the parameter names, and the values are the parameter values.
  128: 
  129: =back
  130: 
  131: It takes optional arguments:
  132: 
  133: =over 4
  134: 
  135: =item * C<description>
  136: 
  137: What the schema does,
  138: used in error messages.
  139: 
  140: =item * C<error_msg>
  141: 
  142: Overrides the default message when something doesn't validate.
  143: 
  144: =item * C<unknown_parameter_handler>
  145: 
  146: This parameter describes what to do when a parameter is given that is not in the schema of valid parameters.
  147: It must be one of C<die>, C<warn>, or C<ignore>.
  148: 
  149: It defaults to C<die> unless C<carp_on_warn> is given, in which case it defaults to C<warn>.
  150: 
  151: =item * C<logger>
  152: 
  153: A logging object that understands messages such as C<error> and C<warn>.
  154: 
  155: =item * C<custom_types>
  156: 
  157: A reference to a hash that defines reusable custom types.
  158: Custom types allow you to define validation rules once and reuse them throughout your schema,
  159: making your validation logic more maintainable and readable.
  160: 
  161: Each custom type is defined as a hash reference containing the same validation rules available for regular parameters
  162: (C<type>, C<min>, C<max>, C<matches>, C<memberof>, C<values>, C<enum>, C<notmemberof>, C<callback>, etc.).
  163: 
  164:   my $custom_types = {
  165:     email => {
  166:       type => 'string',
  167:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/,
  168:       error_msg => 'Invalid email address format'
  169:     }, phone => {
  170:       type => 'string',
  171:       matches => qr/^\+?[1-9]\d{1,14}$/,
  172:       min => 10,
  173:       max => 15
  174:     }, percentage => {
  175:       type => 'number',
  176:       min => 0,
  177:       max => 100
  178:     }, status => {
  179:       type => 'string',
  180:       memberof => ['draft', 'published', 'archived']
  181:     }
  182:   };
  183: 
  184:   my $schema = {
  185:     user_email => { type => 'email' },
  186:     contact_number => { type => 'phone', optional => 1 },
  187:     completion => { type => 'percentage' },
  188:     post_status => { type => 'status' }
  189:   };
  190: 
  191:   my $validated = validate_strict(
  192:     schema => $schema,
  193:     input => $input,
  194:     custom_types => $custom_types
  195:   );
  196: 
  197: Custom types can be extended or overridden in the schema by specifying additional constraints:
  198: 
  199:   my $schema = {
  200:     admin_username => {
  201:       type => 'username',  # Uses custom type definition
  202:       min => 5,            # Overrides custom type's min value
  203:       max => 15            # Overrides custom type's max value
  204:     }
  205:   };
  206: 
  207: Custom types work seamlessly with nested schema, optional parameters, and all other validation features.
  208: 
  209: =back
  210: 
  211: The schema can define the following rules for each parameter:
  212: 
  213: =over 4
  214: 
  215: =item * C<type>
  216: 
  217: The data type of the parameter.
  218: Valid types are C<string>, C<integer>, C<number>, C<float> C<boolean>, C<scalar>, C<scalarref>, C<stringref>, C<hashref>, C<arrayref>, C<object> and C<coderef>.
  219: C<scalar> accepts any plain scalar value (string, number, boolean, etc.) but rejects references (arrayrefs, hashrefs, coderefs, objects).
  220: C<scalarref> accepts a reference to a scalar value (e.g. C<\$var>) but rejects plain scalars, arrayrefs, hashrefs, coderefs, and objects.
  221: C<stringref> accepts a reference to a scalar that contains a plain string (e.g. C<\$str>) and rejects plain scalars, references-to-references, arrayrefs, hashrefs, coderefs, and objects.
  222: The C<min>/C<max> constraints apply to the B<length> (in characters) of the referenced string.
  223: All other string rules (C<matches>, C<nomatch>, C<memberof>, etc.) operate on the dereferenced string value.
  224: The validated return value is the dereferenced plain string.
  225: 
  226: A type can be an arrayref when a parameter could have different types (e.g. a string or an object).
  227: 
  228:   $schema = {
  229:     username => [
  230:       { type => 'string', min => 3, max => 50 },	# Name
  231:       { type => 'integer', 'min' => 1 },	# UID that isn't root
  232:     ]
  233:   };
  234: 
  235: As a shorthand, C<type> itself may be an arrayref of type name strings (a I<union type>)
  236: when all other constraints are shared between the alternatives:
  237: 
  238:   $schema = {
  239:     data => { type => ['string', 'arrayref'] },
  240:     id   => { type => ['string', 'integer'], optional => 1 },
  241:   };
  242: 
  243: This is equivalent to the full array-of-rules form but more concise.
  244: Every other key in the rule hash (C<optional>, C<min>, C<max>, C<matches>, etc.)
  245: is inherited by each candidate type and validated independently against it.
  246: Type names are tried left-to-right; the first match wins and its coercion
  247: (e.g. numeric types) is propagated back to the caller.
  248: If the value fails all candidate types, validation croaks with a message
  249: listing the union members.
  250: 
  251: =item * C<can>
  252: 
  253: The parameter must be an object that understands the method C<can>.
  254: C<can> can be a simple scalar string of a method name,
  255: or an arrayref of a list of method names, all of which must be supported by the object.
  256: 
  257:    $schema = {
  258:      gedcom => { type => object, can => 'get_individual' }
  259:    }
  260: 
  261: =item * C<isa>
  262: 
  263: The parameter must be an object of type C<isa>.
  264: 
  265: =item * C<memberof>
  266: 
  267: The parameter must be a member of the given arrayref.
  268: 
  269:   status => {
  270:     type => 'string',
  271:     memberof => ['draft', 'published', 'archived']
  272:   }
  273: 
  274:   priority => {
  275:     type => 'integer',
  276:     memberof => [1, 2, 3, 4, 5]
  277:   }
  278: 
  279: For string types, the comparison is case-sensitive by default. Use the C<case_sensitive>
  280: flag to control this behavior:
  281: 
  282:   # Case-sensitive (default) - must be exact match
  283:   code => {
  284:     type => 'string',
  285:     memberof => ['ABC', 'DEF', 'GHI']
  286:     # 'abc' will fail
  287:   }
  288: 
  289:   # Case-insensitive - any case accepted
  290:   code => {
  291:     type => 'string',
  292:     memberof => ['ABC', 'DEF', 'GHI'],
  293:     case_sensitive => 0
  294:     # 'abc', 'Abc', 'ABC' all pass, original case preserved
  295:   }
  296: 
  297: For numeric types (C<integer>, C<number>, C<float>), the comparison uses numeric
  298: equality (C<==> operator):
  299: 
  300:   rating => {
  301:     type => 'number',
  302:     memberof => [0.5, 1.0, 1.5, 2.0]
  303:   }
  304: 
  305: Note that C<memberof> cannot be combined with C<min> or C<max> constraints as they
  306: serve conflicting purposes - C<memberof> defines an explicit whitelist while C<min>/C<max>
  307: define ranges.
  308: 
  309: =item * C<enum>
  310: 
  311: Same as C<memberof>.
  312: 
  313: =item * C<values>
  314: 
  315: Same as C<memberof>.
  316: 
  317: =item * C<notmemberof>
  318: 
  319: The parameter must not be a member of the given arrayref (blacklist).
  320: This is the inverse of C<memberof>.
  321: 
  322:   username => {
  323:     type => 'string',
  324:     notmemberof => ['admin', 'root', 'system', 'administrator']
  325:   }
  326: 
  327:   port => {
  328:     type => 'integer',
  329:     notmemberof => [22, 23, 25, 80, 443]  # Reserved ports
  330:   }
  331: 
  332: Like C<memberof>, string comparisons are case-sensitive by default but can be controlled
  333: with the C<case_sensitive> flag:
  334: 
  335:   # Case-sensitive (default)
  336:   username => {
  337:     type => 'string',
  338:     notmemberof => ['Admin', 'Root']
  339:     # 'admin' would pass, 'Admin' would fail
  340:   }
  341: 
  342:   # Case-insensitive
  343:   username => {
  344:     type => 'string',
  345:     notmemberof => ['Admin', 'Root'],
  346:     case_sensitive => 0
  347:     # 'admin', 'ADMIN', 'Admin' all fail
  348:   }
  349: 
  350: The blacklist is checked after any C<transform> rules are applied, allowing you to
  351: normalize input before checking:
  352: 
  353:   username => {
  354:     type => 'string',
  355:     transform => sub { lc($_[0]) },  # Normalize to lowercase
  356:     notmemberof => ['admin', 'root', 'system']
  357:   }
  358: 
  359: C<notmemberof> can be combined with other validation rules:
  360: 
  361:   username => {
  362:     type => 'string',
  363:     notmemberof => ['admin', 'root', 'system'],
  364:     min => 3,
  365:     max => 20,
  366:     matches => qr/^[a-z0-9_]+$/
  367:   }
  368: 
  369: =item * C<case_sensitive>
  370: 
  371: A boolean value indicating whether string comparisons should be case-sensitive.
  372: This flag affects the C<memberof> and C<notmemberof> validation rules.
  373: The default value is C<1> (case-sensitive).
  374: 
  375: When set to C<0>, string comparisons are performed case-insensitively, allowing values
  376: with different casing to match. The original case of the input value is preserved in
  377: the validated output.
  378: 
  379:   # Case-sensitive (default)
  380:   status => {
  381:     type => 'string',
  382:     memberof => ['Draft', 'Published', 'Archived'] # Input 'draft' will fail - must match exact case
  383:   }
  384: 
  385:   # Case-insensitive
  386:   status => {
  387:     type => 'string',
  388:     memberof => ['Draft', 'Published', 'Archived'],
  389:     case_sensitive => 0 # Input 'draft', 'DRAFT', or 'DrAfT' will all pass
  390:   }
  391: 
  392:   country_code => {
  393:     type => 'string',
  394:     memberof => ['US', 'UK', 'CA', 'FR'],
  395:     case_sensitive => 0  # Accept 'us', 'US', 'Us', etc.
  396:   }
  397: 
  398: This flag has no effect on numeric types (C<integer>, C<number>, C<float>) as numbers
  399: do not have case.
  400: 
  401: =item * C<min>/C<minimum>
  402: 
  403: The minimum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  404: 
  405: =item * C<max>
  406: 
  407: The maximum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  408: 
  409: =item * C<matches>
  410: 
  411: A regular expression that the parameter value must match.
  412: Checks all members of arrayrefs.
  413: 
  414: =item * C<nomatch>
  415: 
  416: A regular expression that the parameter value must not match.
  417: Checks all members of arrayrefs.
  418: 
  419: =item * C<position>
  420: 
  421: For routines and methods that take positional args,
  422: this integer value defines which position the argument will be in.
  423: If this is set for all arguments,
  424: C<validate_strict> will return a reference to an array, rather than a reference to a hash.
  425: 
  426: =item * C<regex>
  427: 
  428: Synonym of matches
  429: 
  430: =item * C<description>
  431: 
  432: The description of the rule
  433: 
  434: =item * C<callback>
  435: 
  436: A code reference to a subroutine that performs custom validation logic.
  437: The subroutine should accept the parameter value, the argument list and the schema as arguments and return true if the value is valid, false otherwise.
  438: 
  439: Use this to test more complex examples:
  440: 
  441:   my $schema = {
  442:     even_number => {
  443:       type => 'integer',
  444:       callback => sub { $_[0] % 2 == 0 }
  445:   };
  446: 
  447:   # Specify the arguments for a routine which has a second, optional argument, which, if given, must be less than or equal to the first
  448:   my $schema = {
  449:     first => {
  450:       type => 'integer'
  451:     }, second => {
  452:       type => 'integer',
  453:       optional => 1,
  454:       callback => sub {
  455:         my($value, $args) = @_;
  456: 	# The 'defined' is needed in case 'second' is evaluated before 'first'
  457: 	return (defined($args->{first}) && $value <= $args->{first}) ? 1 : 0
  458:       }
  459:     }
  460:   };
  461: 
  462: =item * C<optional>
  463: 
  464: A boolean value indicating whether the parameter is optional.
  465: If true, the parameter is not required.
  466: If false or omitted, the parameter is required.
  467: 
  468: It can be a reference to a code snippet that will return true or false,
  469: to determine if the parameter is optional or not.
  470: The code will be called with two arguments: the value of the parameter and hash ref of all parameters:
  471: 
  472:   my $schema = {
  473:     optional_field => {
  474:       type => 'string',
  475:       optional => sub {
  476:         my ($value, $all_params) = @_;
  477:         return $all_params->{make_optional} ? 1 : 0;
  478:       }
  479:     },
  480:     make_optional => { type => 'boolean' }
  481:   };
  482: 
  483:   my $result = validate_strict(schema => $schema, input => { make_optional => 1 });
  484: 
  485: If the parameter is not optional, it can be passed an undef value, which will not flag an error.
  486: This is by design.
  487: So this will not say that the required parameter 's' is missing:
  488: 
  489:     validate_strict(
  490:         schema => { s => { type => 'string' } },
  491:         input  => { s => undef },
  492:     );
  493: 
  494: =item * C<default>
  495: 
  496: Populate missing optional parameters with the specified value.
  497: Note that this value is not validated.
  498: 
  499:   username => {
  500:     type => 'string',
  501:     optional => 1,
  502:     default => 'guest'
  503:   }
  504: 
  505: =item * C<element_type>
  506: 
  507: Extends the validation to individual elements of arrays.
  508: 
  509:   tags => {
  510:     type => 'arrayref',
  511:     element_type => 'number',	# Float means the same
  512:     min => 1,	# this is the length of the array, not the min value for each of the numbers. For that, add a C<schema> rule
  513:     max => 5
  514:   }
  515: 
  516: =item * C<error_msg>
  517: 
  518: The custom error message to be used in the event of a validation failure.
  519: 
  520:   age => {
  521:     type => 'integer',
  522:     min => 18,
  523:     error_msg => 'You must be at least 18 years old'
  524:   }
  525: 
  526: =item * C<nullable>
  527: 
  528: Like optional,
  529: though this cannot be a coderef,
  530: only a flag.
  531: 
  532: =item * C<schema>
  533: 
  534: You can validate nested hashrefs and arrayrefs using the C<schema> property:
  535: 
  536:     my $schema = {
  537:         user => {	# 'user' is a hashref
  538:             type => 'hashref',
  539:             schema => {	# Specify what the elements of the hash should be
  540:                 name => { type => 'string' },
  541:                 age => { type => 'integer', min => 0 },
  542:                 hobbies => {	# 'hobbies' is an array ref that this user has
  543:                     type => 'arrayref',
  544:                     schema => { type => 'string' }, # Validate each hobby
  545:                     min => 1 # At least one hobby
  546:                 }
  547:             }
  548:         }, metadata => {
  549:             type => 'hashref',
  550:             schema => {
  551:                 created => { type => 'string' },
  552:                 tags => {
  553:                     type => 'arrayref',
  554:                     schema => {
  555:                         type => 'string',
  556:                         matches => qr/^[a-z]+$/	# Or you can say matches => '^[a-z]+$'
  557:                     }
  558:                 }
  559:             }
  560:         }
  561:     };
  562: 
  563: =item * C<validate>
  564: 
  565: A snippet of code that validates the input.
  566: It's passed the input arguments,
  567: and return a string containing a reason for rejection,
  568: or undef if it's allowed.
  569: 
  570:     my $schema = {
  571:       user => {
  572:         type => 'string',
  573: 	validate => sub {
  574: 	  if($_[0]->{'password'} eq 'bar') {
  575: 	    return undef;
  576: 	  }
  577: 	  return 'Invalid password, try again';
  578: 	}
  579:       }, password => {
  580:          type => 'string'
  581:       }
  582:     };
  583: 
  584: =item * C<transform>
  585: 
  586: A code reference to a subroutine that transforms/sanitizes the parameter value before validation.
  587: The subroutine should accept the parameter value as an argument and return the transformed value.
  588: The transformation is applied before any validation rules are checked, allowing you to normalize
  589: or clean data before it is validated.
  590: 
  591: Common use cases include trimming whitespace, normalizing case, formatting phone numbers,
  592: sanitizing user input, and converting between data formats.
  593: 
  594:   # Simple string transformations
  595:   username => {
  596:     type => 'string',
  597:     transform => sub { lc(trim($_[0])) },  # lowercase and trim
  598:     matches => qr/^[a-z0-9_]+$/
  599:   }
  600: 
  601:   email => {
  602:     type => 'string',
  603:     transform => sub { lc(trim($_[0])) },  # normalize email
  604:     matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  605:   }
  606: 
  607:   # Array transformations
  608:   tags => {
  609:     type => 'arrayref',
  610:     transform => sub { [map { lc($_) } @{$_[0]}] },  # lowercase all elements
  611:     element_type => 'string'
  612:   }
  613: 
  614:   keywords => {
  615:     type => 'arrayref',
  616:     transform => sub {
  617:       my @arr = map { lc(trim($_)) } @{$_[0]};
  618:       my %seen;
  619:       return [grep { !$seen{$_}++ } @arr];  # remove duplicates
  620:     }
  621:   }
  622: 
  623:   # Numeric transformations
  624:   quantity => {
  625:     type => 'integer',
  626:     transform => sub { int($_[0] + 0.5) },  # round to nearest integer
  627:     min => 1
  628:   }
  629: 
  630:   # Sanitization
  631:   slug => {
  632:     type => 'string',
  633:     transform => sub {

  702: =item * C<cross_validation>
  703: 
  704: A reference to a hash that defines validation rules that depend on more than one parameter.
  705: Cross-field validations are performed after all individual parameter validations have passed,
  706: allowing you to enforce business logic that requires checking relationships between different fields.
  707: 
  708: Each cross-validation rule is a key-value pair where the key is a descriptive name for the validation
  709: and the value is a code reference that accepts a hash reference of all validated parameters.
  710: The subroutine should return C<undef> if the validation passes, or an error message string if it fails.
  711: 
  712:   my $schema = {
  713:     password => { type => 'string', min => 8 },
  714:     password_confirm => { type => 'string' }
  715:   };
  716: 
  717:   my $cross_validation = {
  718:     passwords_match => sub {
  719:       my $params = shift;
  720:       return $params->{password} eq $params->{password_confirm}
  721:         ? undef : "Passwords don't match";
  722:     }
  723:   };
  724: 
  725:   my $validated = validate_strict(
  726:     schema => $schema,
  727:     input => $input,
  728:     cross_validation => $cross_validation
  729:   );
  730: 
  731: Common use cases include password confirmation, date range validation, numeric comparisons,
  732: and conditional requirements:
  733: 
  734:   # Date range validation
  735:   my $cross_validation = {
  736:     date_range_valid => sub {
  737:       my $params = shift;
  738:       return $params->{start_date} le $params->{end_date}
  739:         ? undef : "Start date must be before or equal to end date";
  740:     }
  741:   };
  742: 
  743:   # Price range validation
  744:   my $cross_validation = {
  745:     price_range_valid => sub {
  746:       my $params = shift;
  747:       return $params->{min_price} <= $params->{max_price}
  748:         ? undef : "Minimum price must be less than or equal to maximum price";
  749:     }
  750:   };

  752:   # Conditional required field
  753:   my $cross_validation = {
  754:     address_required_for_delivery => sub {
  755:       my $params = shift;
  756:       if ($params->{shipping_method} eq 'delivery' && !$params->{delivery_address}) {
  757:         return "Delivery address is required when shipping method is 'delivery'";
  758:       }

  760:     }
  761:   };

  763: Multiple cross-validations can be defined in the same hash, and they are all checked in order.
  764: If any cross-validation fails, the function will C<croak> with the error message returned by the validation:
  765: 
  766:   my $cross_validation = {

  832: =item * C<relationships>
  833: 
  834: A reference to an array that defines validation rules based on relationships between parameters.
  835: Relationship validations are performed after all individual parameter validations have passed,
  836: but before cross-validations.
  837: 
  838: Each relationship is a hash reference with a C<type> field and additional fields depending on the type:
  839: 
  840: =over 4
  841: 
  842: =item * B<mutually_exclusive>
  843: 
  844: Parameters that cannot be specified together.
  845: 
  846:   relationships => [
  847:     {
  848:       type => 'mutually_exclusive',
  849:       params => ['file', 'content'],
  850:       description => 'Cannot specify both file and content'
  851:     }
  852:   ]
  853: 
  854: =item * B<required_group>
  855: 
  856: At least one parameter from the group must be specified.
  857: 
  858:   relationships => [
  859:     {
  860:       type => 'required_group',
  861:       params => ['id', 'name'],
  862:       logic => 'or',
  863:       description => 'Must specify either id or name'
  864:     }
  865:   ]
  866: 
  867: =item * B<conditional_requirement>
  868: 
  869: If one parameter is specified, another becomes required.
  870: 
  871:   relationships => [
  872:     {
  873:       type => 'conditional_requirement',
  874:       if => 'async',
  875:       then_required => 'callback',
  876:       description => 'When async is specified, callback is required'
  877:     }
  878:   ]
  879: 
  880: =item * B<dependency>
  881: 
  882: One parameter requires another to be present.
  883: 
  884:   relationships => [
  885:     {
  886:       type => 'dependency',
  887:       param => 'port',
  888:       requires => 'host',
  889:       description => 'port requires host to be specified'
  890:     }
  891:   ]
  892: 
  893: =item * B<value_constraint>
  894: 
  895: Specific value requirements between parameters.
  896: 
  897:   relationships => [
  898:     {
  899:       type => 'value_constraint',
  900:       if => 'ssl',
  901:       then => 'port',
  902:       operator => '==',
  903:       value => 443,
  904:       description => 'When ssl is specified, port must equal 443'
  905:     }
  906:   ]
  907: 
  908: =item * B<value_conditional>
  909: 
  910: Parameter required when another has a specific value.
  911: 
  912:   relationships => [
  913:     {
  914:       type => 'value_conditional',
  915:       if => 'mode',
  916:       equals => 'secure',
  917:       then_required => 'key',
  918:       description => "When mode equals 'secure', key is required"
  919:     }
  920:   ]
  921: 
  922: =back
  923: 
  924: If a parameter is optional and its value is C<undef>,
  925: validation will be skipped for that parameter.
  926: 
  927: If the validation fails, the function will C<croak> with an error message describing the validation failure.
  928: 
  929: If the validation is successful, the function will return a reference to a new hash containing the validated and (where applicable) coerced parameters.  Integer and number parameters will be coerced to their respective types.
  930: 
  931: The C<description> field is optional but recommended for clearer error messages.
  932: 
  933: =back
