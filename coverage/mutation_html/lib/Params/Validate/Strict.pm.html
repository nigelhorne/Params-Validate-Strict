   83: =head2 validate_strict
   84: 
   85: Validates a set of parameters against a schema.
   86: 
   87: This function takes two mandatory arguments:
   88: 
   89: =over 4
   90: 
   91: =item * C<schema> || C<members>
   92: 
   93: A reference to a hash that defines the validation rules for each parameter.
   94: The keys of the hash are the parameter names, and the values are either a string representing the parameter type or a reference to a hash containing more detailed rules.
   95: 
   96: As an alternative the schema may be supplied as an B<arrayref of parameter
   97: hashrefs>, where every element describes one parameter and carries a mandatory
   98: C<name> key:
   99: 
  100:   $schema = [
  101:     { name => 'username', type => 'string', min => 3, max => 50 },
  102:     { name => 'age',      type => 'integer', min => 0, max => 150 },
  103:     { name => 'role',     type => 'string', optional => 1, default => 'user' },
  104:   ];
  105: 
  106: The arrayref form is normalised to the standard hashref form before any further
  107: processing.  It is particularly useful when declaration order matters (e.g.
  108: for positional or mixed calling conventions used by some CPAN modules).  The
  109: C<name> key is consumed during normalisation and does not appear as a
  110: validation rule.
  111: 
  112: For some sort of compatibility with L<Data::Processor>,
  113: it is possible to wrap the schema within a hash like this:
  114: 
  115:   $schema = {
  116:     description => 'Describe what this schema does',
  117:     error_msg => 'An error message',
  118:     schema => {
  119:       # ... schema goes here
  120:     }
  121:   }
  122: 
  123: =item * C<args> || C<input>
  124: 
  125: A reference to a hash containing the parameters to be validated.
  126: The keys of the hash are the parameter names, and the values are the parameter values.
  127: 
  128: =back
  129: 
  130: It takes optional arguments:
  131: 
  132: =over 4
  133: 
  134: =item * C<description>
  135: 
  136: What the schema does,
  137: used in error messages.
  138: 
  139: =item * C<error_msg>
  140: 
  141: Overrides the default message when something doesn't validate.
  142: 
  143: =item * C<unknown_parameter_handler>
  144: 
  145: This parameter describes what to do when a parameter is given that is not in the schema of valid parameters.
  146: It must be one of C<die>, C<warn>, or C<ignore>.
  147: 
  148: It defaults to C<die> unless C<carp_on_warn> is given, in which case it defaults to C<warn>.
  149: 
  150: =item * C<logger>
  151: 
  152: A logging object that understands messages such as C<error> and C<warn>.
  153: 
  154: =item * C<custom_types>
  155: 
  156: A reference to a hash that defines reusable custom types.
  157: Custom types allow you to define validation rules once and reuse them throughout your schema,
  158: making your validation logic more maintainable and readable.
  159: 
  160: Each custom type is defined as a hash reference containing the same validation rules available for regular parameters
  161: (C<type>, C<min>, C<max>, C<matches>, C<memberof>, C<values>, C<enum>, C<notmemberof>, C<callback>, etc.).
  162: 
  163:   my $custom_types = {
  164:     email => {
  165:       type => 'string',
  166:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/,
  167:       error_msg => 'Invalid email address format'
  168:     }, phone => {
  169:       type => 'string',
  170:       matches => qr/^\+?[1-9]\d{1,14}$/,
  171:       min => 10,
  172:       max => 15
  173:     }, percentage => {
  174:       type => 'number',
  175:       min => 0,
  176:       max => 100
  177:     }, status => {
  178:       type => 'string',
  179:       memberof => ['draft', 'published', 'archived']
  180:     }
  181:   };
  182: 
  183:   my $schema = {
  184:     user_email => { type => 'email' },
  185:     contact_number => { type => 'phone', optional => 1 },
  186:     completion => { type => 'percentage' },
  187:     post_status => { type => 'status' }
  188:   };
  189: 
  190:   my $validated = validate_strict(
  191:     schema => $schema,
  192:     input => $input,
  193:     custom_types => $custom_types
  194:   );
  195: 
  196: Custom types can be extended or overridden in the schema by specifying additional constraints:
  197: 
  198:   my $schema = {
  199:     admin_username => {
  200:       type => 'username',  # Uses custom type definition
  201:       min => 5,            # Overrides custom type's min value
  202:       max => 15            # Overrides custom type's max value
  203:     }
  204:   };
  205: 
  206: Custom types work seamlessly with nested schema, optional parameters, and all other validation features.
  207: 
  208: =back
  209: 
  210: The schema can define the following rules for each parameter:
  211: 
  212: =over 4
  213: 
  214: =item * C<type>
  215: 
  216: The data type of the parameter.
  217: Valid types are C<string>, C<integer>, C<number>, C<float> C<boolean>, C<scalar>, C<scalarref>, C<hashref>, C<arrayref>, C<object> and C<coderef>.
  218: C<scalar> accepts any plain scalar value (string, number, boolean, etc.) but rejects references (arrayrefs, hashrefs, coderefs, objects).
  219: C<scalarref> accepts a reference to a scalar value (e.g. C<\$var>) but rejects plain scalars, arrayrefs, hashrefs, coderefs, and objects.
  220: 
  221: A type can be an arrayref when a parameter could have different types (e.g. a string or an object).
  222: 
  223:   $schema = {
  224:     username => [
  225:       { type => 'string', min => 3, max => 50 },	# Name
  226:       { type => 'integer', 'min' => 1 },	# UID that isn't root
  227:     ]
  228:   };
  229: 
  230: As a shorthand, C<type> itself may be an arrayref of type name strings (a I<union type>)
  231: when all other constraints are shared between the alternatives:
  232: 
  233:   $schema = {
  234:     data => { type => ['string', 'arrayref'] },
  235:     id   => { type => ['string', 'integer'], optional => 1 },
  236:   };
  237: 
  238: This is equivalent to the full array-of-rules form but more concise.
  239: Every other key in the rule hash (C<optional>, C<min>, C<max>, C<matches>, etc.)
  240: is inherited by each candidate type and validated independently against it.
  241: Type names are tried left-to-right; the first match wins and its coercion
  242: (e.g. numeric types) is propagated back to the caller.
  243: If the value fails all candidate types, validation croaks with a message
  244: listing the union members.
  245: 
  246: =item * C<can>
  247: 
  248: The parameter must be an object that understands the method C<can>.
  249: C<can> can be a simple scalar string of a method name,
  250: or an arrayref of a list of method names, all of which must be supported by the object.
  251: 
  252:    $schema = {
  253:      gedcom => { type => object, can => 'get_individual' }
  254:    }
  255: 
  256: =item * C<isa>
  257: 
  258: The parameter must be an object of type C<isa>.
  259: 
  260: =item * C<memberof>
  261: 
  262: The parameter must be a member of the given arrayref.
  263: 
  264:   status => {
  265:     type => 'string',
  266:     memberof => ['draft', 'published', 'archived']
  267:   }
  268: 
  269:   priority => {
  270:     type => 'integer',
  271:     memberof => [1, 2, 3, 4, 5]
  272:   }
  273: 
  274: For string types, the comparison is case-sensitive by default. Use the C<case_sensitive>
  275: flag to control this behavior:
  276: 
  277:   # Case-sensitive (default) - must be exact match
  278:   code => {
  279:     type => 'string',
  280:     memberof => ['ABC', 'DEF', 'GHI']
  281:     # 'abc' will fail
  282:   }
  283: 
  284:   # Case-insensitive - any case accepted
  285:   code => {
  286:     type => 'string',
  287:     memberof => ['ABC', 'DEF', 'GHI'],
  288:     case_sensitive => 0
  289:     # 'abc', 'Abc', 'ABC' all pass, original case preserved
  290:   }
  291: 
  292: For numeric types (C<integer>, C<number>, C<float>), the comparison uses numeric
  293: equality (C<==> operator):
  294: 
  295:   rating => {
  296:     type => 'number',
  297:     memberof => [0.5, 1.0, 1.5, 2.0]
  298:   }
  299: 
  300: Note that C<memberof> cannot be combined with C<min> or C<max> constraints as they
  301: serve conflicting purposes - C<memberof> defines an explicit whitelist while C<min>/C<max>
  302: define ranges.
  303: 
  304: =item * C<enum>
  305: 
  306: Same as C<memberof>.
  307: 
  308: =item * C<values>
  309: 
  310: Same as C<memberof>.
  311: 
  312: =item * C<notmemberof>
  313: 
  314: The parameter must not be a member of the given arrayref (blacklist).
  315: This is the inverse of C<memberof>.
  316: 
  317:   username => {
  318:     type => 'string',
  319:     notmemberof => ['admin', 'root', 'system', 'administrator']
  320:   }
  321: 
  322:   port => {
  323:     type => 'integer',
  324:     notmemberof => [22, 23, 25, 80, 443]  # Reserved ports
  325:   }
  326: 
  327: Like C<memberof>, string comparisons are case-sensitive by default but can be controlled
  328: with the C<case_sensitive> flag:
  329: 
  330:   # Case-sensitive (default)
  331:   username => {
  332:     type => 'string',
  333:     notmemberof => ['Admin', 'Root']
  334:     # 'admin' would pass, 'Admin' would fail
  335:   }
  336: 
  337:   # Case-insensitive
  338:   username => {
  339:     type => 'string',
  340:     notmemberof => ['Admin', 'Root'],
  341:     case_sensitive => 0
  342:     # 'admin', 'ADMIN', 'Admin' all fail
  343:   }
  344: 
  345: The blacklist is checked after any C<transform> rules are applied, allowing you to
  346: normalize input before checking:
  347: 
  348:   username => {
  349:     type => 'string',
  350:     transform => sub { lc($_[0]) },  # Normalize to lowercase
  351:     notmemberof => ['admin', 'root', 'system']
  352:   }
  353: 
  354: C<notmemberof> can be combined with other validation rules:
  355: 
  356:   username => {
  357:     type => 'string',
  358:     notmemberof => ['admin', 'root', 'system'],
  359:     min => 3,
  360:     max => 20,
  361:     matches => qr/^[a-z0-9_]+$/
  362:   }
  363: 
  364: =item * C<case_sensitive>
  365: 
  366: A boolean value indicating whether string comparisons should be case-sensitive.
  367: This flag affects the C<memberof> and C<notmemberof> validation rules.
  368: The default value is C<1> (case-sensitive).
  369: 
  370: When set to C<0>, string comparisons are performed case-insensitively, allowing values
  371: with different casing to match. The original case of the input value is preserved in
  372: the validated output.
  373: 
  374:   # Case-sensitive (default)
  375:   status => {
  376:     type => 'string',
  377:     memberof => ['Draft', 'Published', 'Archived'] # Input 'draft' will fail - must match exact case
  378:   }
  379: 
  380:   # Case-insensitive
  381:   status => {
  382:     type => 'string',
  383:     memberof => ['Draft', 'Published', 'Archived'],
  384:     case_sensitive => 0 # Input 'draft', 'DRAFT', or 'DrAfT' will all pass
  385:   }
  386: 
  387:   country_code => {
  388:     type => 'string',
  389:     memberof => ['US', 'UK', 'CA', 'FR'],
  390:     case_sensitive => 0  # Accept 'us', 'US', 'Us', etc.
  391:   }
  392: 
  393: This flag has no effect on numeric types (C<integer>, C<number>, C<float>) as numbers
  394: do not have case.
  395: 
  396: =item * C<min>/C<minimum>
  397: 
  398: The minimum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  399: 
  400: =item * C<max>
  401: 
  402: The maximum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  403: 
  404: =item * C<matches>
  405: 
  406: A regular expression that the parameter value must match.
  407: Checks all members of arrayrefs.
  408: 
  409: =item * C<nomatch>
  410: 
  411: A regular expression that the parameter value must not match.
  412: Checks all members of arrayrefs.
  413: 
  414: =item * C<position>
  415: 
  416: For routines and methods that take positional args,
  417: this integer value defines which position the argument will be in.
  418: If this is set for all arguments,
  419: C<validate_strict> will return a reference to an array, rather than a reference to a hash.
  420: 
  421: =item * C<regex>
  422: 
  423: Synonym of matches
  424: 
  425: =item * C<description>
  426: 
  427: The description of the rule
  428: 
  429: =item * C<callback>
  430: 
  431: A code reference to a subroutine that performs custom validation logic.
  432: The subroutine should accept the parameter value, the argument list and the schema as arguments and return true if the value is valid, false otherwise.
  433: 
  434: Use this to test more complex examples:
  435: 
  436:   my $schema = {
  437:     even_number => {
  438:       type => 'integer',
  439:       callback => sub { $_[0] % 2 == 0 }
  440:   };
  441: 
  442:   # Specify the arguments for a routine which has a second, optional argument, which, if given, must be less than or equal to the first
  443:   my $schema = {
  444:     first => {
  445:       type => 'integer'
  446:     }, second => {
  447:       type => 'integer',
  448:       optional => 1,
  449:       callback => sub {
  450:         my($value, $args) = @_;
  451: 	# The 'defined' is needed in case 'second' is evaluated before 'first'
  452: 	return (defined($args->{first}) && $value <= $args->{first}) ? 1 : 0
  453:       }
  454:     }
  455:   };
  456: 
  457: =item * C<optional>
  458: 
  459: A boolean value indicating whether the parameter is optional.
  460: If true, the parameter is not required.
  461: If false or omitted, the parameter is required.
  462: 
  463: It can be a reference to a code snippet that will return true or false,
  464: to determine if the parameter is optional or not.
  465: The code will be called with two arguments: the value of the parameter and hash ref of all parameters:
  466: 
  467:   my $schema = {
  468:     optional_field => {
  469:       type => 'string',
  470:       optional => sub {
  471:         my ($value, $all_params) = @_;
  472:         return $all_params->{make_optional} ? 1 : 0;
  473:       }
  474:     },
  475:     make_optional => { type => 'boolean' }
  476:   };
  477: 
  478:   my $result = validate_strict(schema => $schema, input => { make_optional => 1 });
  479: 
  480: If the parameter is not optional, it can be passed an undef value, which will not flag an error.
  481: This is by design.
  482: So this will not say that the required parameter 's' is missing:
  483: 
  484:     validate_strict(
  485:         schema => { s => { type => 'string' } },
  486:         input  => { s => undef },
  487:     );
  488: 
  489: =item * C<default>
  490: 
  491: Populate missing optional parameters with the specified value.
  492: Note that this value is not validated.
  493: 
  494:   username => {
  495:     type => 'string',
  496:     optional => 1,
  497:     default => 'guest'
  498:   }
  499: 
  500: =item * C<element_type>
  501: 
  502: Extends the validation to individual elements of arrays.
  503: 
  504:   tags => {
  505:     type => 'arrayref',
  506:     element_type => 'number',	# Float means the same
  507:     min => 1,	# this is the length of the array, not the min value for each of the numbers. For that, add a C<schema> rule
  508:     max => 5
  509:   }
  510: 
  511: =item * C<error_msg>
  512: 
  513: The custom error message to be used in the event of a validation failure.
  514: 
  515:   age => {
  516:     type => 'integer',
  517:     min => 18,
  518:     error_msg => 'You must be at least 18 years old'
  519:   }
  520: 
  521: =item * C<nullable>
  522: 
  523: Like optional,
  524: though this cannot be a coderef,
  525: only a flag.
  526: 
  527: =item * C<schema>
  528: 
  529: You can validate nested hashrefs and arrayrefs using the C<schema> property:
  530: 
  531:     my $schema = {
  532:         user => {	# 'user' is a hashref
  533:             type => 'hashref',
  534:             schema => {	# Specify what the elements of the hash should be
  535:                 name => { type => 'string' },
  536:                 age => { type => 'integer', min => 0 },
  537:                 hobbies => {	# 'hobbies' is an array ref that this user has
  538:                     type => 'arrayref',
  539:                     schema => { type => 'string' }, # Validate each hobby
  540:                     min => 1 # At least one hobby
  541:                 }
  542:             }
  543:         }, metadata => {
  544:             type => 'hashref',
  545:             schema => {
  546:                 created => { type => 'string' },
  547:                 tags => {
  548:                     type => 'arrayref',
  549:                     schema => {
  550:                         type => 'string',
  551:                         matches => qr/^[a-z]+$/	# Or you can say matches => '^[a-z]+$'
  552:                     }
  553:                 }
  554:             }
  555:         }
  556:     };
  557: 
  558: =item * C<validate>
  559: 
  560: A snippet of code that validates the input.
  561: It's passed the input arguments,
  562: and return a string containing a reason for rejection,
  563: or undef if it's allowed.
  564: 
  565:     my $schema = {
  566:       user => {
  567:         type => 'string',
  568: 	validate => sub {
  569: 	  if($_[0]->{'password'} eq 'bar') {
  570: 	    return undef;
  571: 	  }
  572: 	  return 'Invalid password, try again';
  573: 	}
  574:       }, password => {
  575:          type => 'string'
  576:       }
  577:     };
  578: 
  579: =item * C<transform>
  580: 
  581: A code reference to a subroutine that transforms/sanitizes the parameter value before validation.
  582: The subroutine should accept the parameter value as an argument and return the transformed value.
  583: The transformation is applied before any validation rules are checked, allowing you to normalize
  584: or clean data before it is validated.
  585: 
  586: Common use cases include trimming whitespace, normalizing case, formatting phone numbers,
  587: sanitizing user input, and converting between data formats.
  588: 
  589:   # Simple string transformations
  590:   username => {
  591:     type => 'string',
  592:     transform => sub { lc(trim($_[0])) },  # lowercase and trim
  593:     matches => qr/^[a-z0-9_]+$/
  594:   }
  595: 
  596:   email => {
  597:     type => 'string',
  598:     transform => sub { lc(trim($_[0])) },  # normalize email
  599:     matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  600:   }
  601: 
  602:   # Array transformations
  603:   tags => {
  604:     type => 'arrayref',
  605:     transform => sub { [map { lc($_) } @{$_[0]}] },  # lowercase all elements
  606:     element_type => 'string'
  607:   }
  608: 
  609:   keywords => {
  610:     type => 'arrayref',
  611:     transform => sub {
  612:       my @arr = map { lc(trim($_)) } @{$_[0]};
  613:       my %seen;
  614:       return [grep { !$seen{$_}++ } @arr];  # remove duplicates
  615:     }
  616:   }
  617: 
  618:   # Numeric transformations
  619:   quantity => {
  620:     type => 'integer',
  621:     transform => sub { int($_[0] + 0.5) },  # round to nearest integer
  622:     min => 1
  623:   }
  624: 
  625:   # Sanitization
  626:   slug => {
  627:     type => 'string',
  628:     transform => sub {
  629:       my $str = lc(trim($_[0]));
  630:       $str =~ s/[^\w\s-]//g;  # remove special characters
  631:       $str =~ s/\s+/-/g;      # replace spaces with hyphens
  632:       return $str;
  633:     },
  634:     matches => qr/^[a-z0-9-]+$/
  635:   }
  636: 
  637:   phone => {
  638:     type => 'string',
  639:     transform => sub {
  640:       my $str = $_[0];
  641:       $str =~ s/\D//g;  # remove all non-digits
  642:       return $str;
  643:     },

  652:   user => {

  654:     schema => {

  657:         transform => sub { trim($_[0]) }

  659:         type => 'string',
  660:         transform => sub { lc(trim($_[0])) }

  676: subsequent validation rules will check against. If a transformation might fail, ensure it
  677: handles edge cases appropriately.
  678: It is the responsibility of the transformer to ensure that the type of the returned value is correct,
  679: since that is what will be validated.

  697: =item * C<cross_validation>
  698: 
  699: A reference to a hash that defines validation rules that depend on more than one parameter.
  700: Cross-field validations are performed after all individual parameter validations have passed,
  701: allowing you to enforce business logic that requires checking relationships between different fields.
  702: 
  703: Each cross-validation rule is a key-value pair where the key is a descriptive name for the validation
  704: and the value is a code reference that accepts a hash reference of all validated parameters.
  705: The subroutine should return C<undef> if the validation passes, or an error message string if it fails.
  706: 
  707:   my $schema = {
  708:     password => { type => 'string', min => 8 },
  709:     password_confirm => { type => 'string' }
  710:   };
  711: 
  712:   my $cross_validation = {
  713:     passwords_match => sub {
  714:       my $params = shift;
  715:       return $params->{password} eq $params->{password_confirm}
  716:         ? undef : "Passwords don't match";
  717:     }
  718:   };
  719: 
  720:   my $validated = validate_strict(
  721:     schema => $schema,
  722:     input => $input,
  723:     cross_validation => $cross_validation
  724:   );
  725: 
  726: Common use cases include password confirmation, date range validation, numeric comparisons,
  727: and conditional requirements:
  728: 
  729:   # Date range validation
  730:   my $cross_validation = {
  731:     date_range_valid => sub {
  732:       my $params = shift;
  733:       return $params->{start_date} le $params->{end_date}
  734:         ? undef : "Start date must be before or equal to end date";
  735:     }
  736:   };
  737: 
  738:   # Price range validation
  739:   my $cross_validation = {
  740:     price_range_valid => sub {
  741:       my $params = shift;
  742:       return $params->{min_price} <= $params->{max_price}
  743:         ? undef : "Minimum price must be less than or equal to maximum price";
  744:     }
  745:   };
  746: 
  747:   # Conditional required field
  748:   my $cross_validation = {
  749:     address_required_for_delivery => sub {
  750:       my $params = shift;
  751:       if ($params->{shipping_method} eq 'delivery' && !$params->{delivery_address}) {
  752:         return "Delivery address is required when shipping method is 'delivery'";
  753:       }
  754:       return undef;
  755:     }
  756:   };

  758: Multiple cross-validations can be defined in the same hash, and they are all checked in order.
  759: If any cross-validation fails, the function will C<croak> with the error message returned by the validation:
  760: 
  761:   my $cross_validation = {
  762:     passwords_match => sub {
  763:       my $params = shift;
  764:       return $params->{password} eq $params->{password_confirm}

  779:   };

  782: so you can rely on the data being in the correct format and type:

  827: =item * C<relationships>
  828: 
  829: A reference to an array that defines validation rules based on relationships between parameters.
  830: Relationship validations are performed after all individual parameter validations have passed,
  831: but before cross-validations.
  832: 
  833: Each relationship is a hash reference with a C<type> field and additional fields depending on the type:
  834: 
  835: =over 4
  836: 
  837: =item * B<mutually_exclusive>
  838: 
  839: Parameters that cannot be specified together.
  840: 
  841:   relationships => [
  842:     {
  843:       type => 'mutually_exclusive',
  844:       params => ['file', 'content'],
  845:       description => 'Cannot specify both file and content'
  846:     }
  847:   ]
  848: 
  849: =item * B<required_group>
  850: 
  851: At least one parameter from the group must be specified.
  852: 
  853:   relationships => [
  854:     {
  855:       type => 'required_group',
  856:       params => ['id', 'name'],
  857:       logic => 'or',
  858:       description => 'Must specify either id or name'
  859:     }
  860:   ]
  861: 
  862: =item * B<conditional_requirement>
  863: 
  864: If one parameter is specified, another becomes required.
  865: 
  866:   relationships => [
  867:     {
  868:       type => 'conditional_requirement',
  869:       if => 'async',
  870:       then_required => 'callback',
  871:       description => 'When async is specified, callback is required'
  872:     }
  873:   ]
  874: 
  875: =item * B<dependency>
  876: 
  877: One parameter requires another to be present.
  878: 
  879:   relationships => [
  880:     {
  881:       type => 'dependency',
  882:       param => 'port',
  883:       requires => 'host',
  884:       description => 'port requires host to be specified'
  885:     }
  886:   ]
  887: 
  888: =item * B<value_constraint>
  889: 
  890: Specific value requirements between parameters.
  891: 
  892:   relationships => [
  893:     {
  894:       type => 'value_constraint',
  895:       if => 'ssl',
  896:       then => 'port',
  897:       operator => '==',
  898:       value => 443,
  899:       description => 'When ssl is specified, port must equal 443'
  900:     }
  901:   ]
  902: 
  903: =item * B<value_conditional>
  904: 
  905: Parameter required when another has a specific value.
  906: 
  907:   relationships => [
  908:     {
  909:       type => 'value_conditional',
  910:       if => 'mode',
  911:       equals => 'secure',
  912:       then_required => 'key',
  913:       description => "When mode equals 'secure', key is required"
  914:     }
  915:   ]
  916: 
  917: =back
  918: 
  919: If a parameter is optional and its value is C<undef>,
  920: validation will be skipped for that parameter.
  921: 
  922: If the validation fails, the function will C<croak> with an error message describing the validation failure.
  923: 
  924: If the validation is successful, the function will return a reference to a new hash containing the validated and (where applicable) coerced parameters.  Integer and number parameters will be coerced to their respective types.
  925: 
  926: The C<description> field is optional but recommended for clearer error messages.
  927: 
  928: =back
