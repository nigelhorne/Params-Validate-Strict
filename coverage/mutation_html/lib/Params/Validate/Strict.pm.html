   83: =head2 validate_strict
   84: 
   85: Validates a set of parameters against a schema.
   86: 
   87: This function takes two mandatory arguments:
   88: 
   89: =over 4
   90: 
   91: =item * C<schema> || C<members>
   92: 
   93: A reference to a hash that defines the validation rules for each parameter.
   94: The keys of the hash are the parameter names, and the values are either a string representing the parameter type or a reference to a hash containing more detailed rules.
   95: 
   96: As an alternative the schema may be supplied as an B<arrayref of parameter
   97: hashrefs>, where every element describes one parameter and carries a mandatory
   98: C<name> key:
   99: 
  100:   $schema = [
  101:     { name => 'username', type => 'string', min => 3, max => 50 },
  102:     { name => 'age',      type => 'integer', min => 0, max => 150 },
  103:     { name => 'role',     type => 'string', optional => 1, default => 'user' },
  104:   ];
  105: 
  106: The arrayref form is normalised to the standard hashref form before any further
  107: processing.  It is particularly useful when declaration order matters (e.g.
  108: for positional or mixed calling conventions used by some CPAN modules).  The
  109: C<name> key is consumed during normalisation and does not appear as a
  110: validation rule.
  111: 
  112: For some sort of compatibility with L<Data::Processor>,
  113: it is possible to wrap the schema within a hash like this:
  114: 
  115:   $schema = {
  116:     description => 'Describe what this schema does',
  117:     error_msg => 'An error message',
  118:     schema => {
  119:       # ... schema goes here
  120:     }
  121:   }
  122: 
  123: =item * C<args> || C<input>
  124: 
  125: A reference to a hash containing the parameters to be validated.
  126: The keys of the hash are the parameter names, and the values are the parameter values.
  127: 
  128: =back
  129: 
  130: It takes optional arguments:
  131: 
  132: =over 4
  133: 
  134: =item * C<description>
  135: 
  136: What the schema does,
  137: used in error messages.
  138: 
  139: =item * C<error_msg>
  140: 
  141: Overrides the default message when something doesn't validate.
  142: 
  143: =item * C<unknown_parameter_handler>
  144: 
  145: This parameter describes what to do when a parameter is given that is not in the schema of valid parameters.
  146: It must be one of C<die>, C<warn>, or C<ignore>.
  147: 
  148: It defaults to C<die> unless C<carp_on_warn> is given, in which case it defaults to C<warn>.
  149: 
  150: =item * C<logger>
  151: 
  152: A logging object that understands messages such as C<error> and C<warn>.
  153: 
  154: =item * C<custom_types>
  155: 
  156: A reference to a hash that defines reusable custom types.
  157: Custom types allow you to define validation rules once and reuse them throughout your schema,
  158: making your validation logic more maintainable and readable.
  159: 
  160: Each custom type is defined as a hash reference containing the same validation rules available for regular parameters
  161: (C<type>, C<min>, C<max>, C<matches>, C<memberof>, C<values>, C<enum>, C<notmemberof>, C<callback>, etc.).
  162: 
  163:   my $custom_types = {
  164:     email => {
  165:       type => 'string',
  166:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/,
  167:       error_msg => 'Invalid email address format'
  168:     }, phone => {
  169:       type => 'string',
  170:       matches => qr/^\+?[1-9]\d{1,14}$/,
  171:       min => 10,
  172:       max => 15
  173:     }, percentage => {
  174:       type => 'number',
  175:       min => 0,
  176:       max => 100
  177:     }, status => {
  178:       type => 'string',
  179:       memberof => ['draft', 'published', 'archived']
  180:     }
  181:   };
  182: 
  183:   my $schema = {
  184:     user_email => { type => 'email' },
  185:     contact_number => { type => 'phone', optional => 1 },
  186:     completion => { type => 'percentage' },
  187:     post_status => { type => 'status' }
  188:   };
  189: 
  190:   my $validated = validate_strict(
  191:     schema => $schema,
  192:     input => $input,
  193:     custom_types => $custom_types
  194:   );
  195: 
  196: Custom types can be extended or overridden in the schema by specifying additional constraints:
  197: 
  198:   my $schema = {
  199:     admin_username => {
  200:       type => 'username',  # Uses custom type definition
  201:       min => 5,            # Overrides custom type's min value
  202:       max => 15            # Overrides custom type's max value
  203:     }
  204:   };
  205: 
  206: Custom types work seamlessly with nested schema, optional parameters, and all other validation features.
  207: 
  208: =back
  209: 
  210: The schema can define the following rules for each parameter:
  211: 
  212: =over 4
  213: 
  214: =item * C<type>
  215: 
  216: The data type of the parameter.
  217: Valid types are C<string>, C<integer>, C<number>, C<float> C<boolean>, C<scalar>, C<hashref>, C<arrayref>, C<object> and C<coderef>.
  218: C<scalar> accepts any plain scalar value (string, number, boolean, etc.) but rejects references (arrayrefs, hashrefs, coderefs, objects).
  219: 
  220: A type can be an arrayref when a parameter could have different types (e.g. a string or an object).
  221: 
  222:   $schema = {
  223:     username => [
  224:       { type => 'string', min => 3, max => 50 },	# Name
  225:       { type => 'integer', 'min' => 1 },	# UID that isn't root
  226:     ]
  227:   };
  228: 
  229: As a shorthand, C<type> itself may be an arrayref of type name strings (a I<union type>)
  230: when all other constraints are shared between the alternatives:
  231: 
  232:   $schema = {
  233:     data => { type => ['string', 'arrayref'] },
  234:     id   => { type => ['string', 'integer'], optional => 1 },
  235:   };
  236: 
  237: This is equivalent to the full array-of-rules form but more concise.
  238: Every other key in the rule hash (C<optional>, C<min>, C<max>, C<matches>, etc.)
  239: is inherited by each candidate type and validated independently against it.
  240: Type names are tried left-to-right; the first match wins and its coercion
  241: (e.g. numeric types) is propagated back to the caller.
  242: If the value fails all candidate types, validation croaks with a message
  243: listing the union members.
  244: 
  245: =item * C<can>
  246: 
  247: The parameter must be an object that understands the method C<can>.
  248: C<can> can be a simple scalar string of a method name,
  249: or an arrayref of a list of method names, all of which must be supported by the object.
  250: 
  251:    $schema = {
  252:      gedcom => { type => object, can => 'get_individual' }
  253:    }
  254: 
  255: =item * C<isa>
  256: 
  257: The parameter must be an object of type C<isa>.
  258: 
  259: =item * C<memberof>
  260: 
  261: The parameter must be a member of the given arrayref.
  262: 
  263:   status => {
  264:     type => 'string',
  265:     memberof => ['draft', 'published', 'archived']
  266:   }
  267: 
  268:   priority => {
  269:     type => 'integer',
  270:     memberof => [1, 2, 3, 4, 5]
  271:   }
  272: 
  273: For string types, the comparison is case-sensitive by default. Use the C<case_sensitive>
  274: flag to control this behavior:
  275: 
  276:   # Case-sensitive (default) - must be exact match
  277:   code => {
  278:     type => 'string',
  279:     memberof => ['ABC', 'DEF', 'GHI']
  280:     # 'abc' will fail
  281:   }
  282: 
  283:   # Case-insensitive - any case accepted
  284:   code => {
  285:     type => 'string',
  286:     memberof => ['ABC', 'DEF', 'GHI'],
  287:     case_sensitive => 0
  288:     # 'abc', 'Abc', 'ABC' all pass, original case preserved
  289:   }
  290: 
  291: For numeric types (C<integer>, C<number>, C<float>), the comparison uses numeric
  292: equality (C<==> operator):
  293: 
  294:   rating => {
  295:     type => 'number',
  296:     memberof => [0.5, 1.0, 1.5, 2.0]
  297:   }
  298: 
  299: Note that C<memberof> cannot be combined with C<min> or C<max> constraints as they
  300: serve conflicting purposes - C<memberof> defines an explicit whitelist while C<min>/C<max>
  301: define ranges.
  302: 
  303: =item * C<enum>
  304: 
  305: Same as C<memberof>.
  306: 
  307: =item * C<values>
  308: 
  309: Same as C<memberof>.
  310: 
  311: =item * C<notmemberof>
  312: 
  313: The parameter must not be a member of the given arrayref (blacklist).
  314: This is the inverse of C<memberof>.
  315: 
  316:   username => {
  317:     type => 'string',
  318:     notmemberof => ['admin', 'root', 'system', 'administrator']
  319:   }
  320: 
  321:   port => {
  322:     type => 'integer',
  323:     notmemberof => [22, 23, 25, 80, 443]  # Reserved ports
  324:   }
  325: 
  326: Like C<memberof>, string comparisons are case-sensitive by default but can be controlled
  327: with the C<case_sensitive> flag:
  328: 
  329:   # Case-sensitive (default)
  330:   username => {
  331:     type => 'string',
  332:     notmemberof => ['Admin', 'Root']
  333:     # 'admin' would pass, 'Admin' would fail
  334:   }
  335: 
  336:   # Case-insensitive
  337:   username => {
  338:     type => 'string',
  339:     notmemberof => ['Admin', 'Root'],
  340:     case_sensitive => 0
  341:     # 'admin', 'ADMIN', 'Admin' all fail
  342:   }
  343: 
  344: The blacklist is checked after any C<transform> rules are applied, allowing you to
  345: normalize input before checking:
  346: 
  347:   username => {
  348:     type => 'string',
  349:     transform => sub { lc($_[0]) },  # Normalize to lowercase
  350:     notmemberof => ['admin', 'root', 'system']
  351:   }
  352: 
  353: C<notmemberof> can be combined with other validation rules:
  354: 
  355:   username => {
  356:     type => 'string',
  357:     notmemberof => ['admin', 'root', 'system'],
  358:     min => 3,
  359:     max => 20,
  360:     matches => qr/^[a-z0-9_]+$/
  361:   }
  362: 
  363: =item * C<case_sensitive>
  364: 
  365: A boolean value indicating whether string comparisons should be case-sensitive.
  366: This flag affects the C<memberof> and C<notmemberof> validation rules.
  367: The default value is C<1> (case-sensitive).
  368: 
  369: When set to C<0>, string comparisons are performed case-insensitively, allowing values
  370: with different casing to match. The original case of the input value is preserved in
  371: the validated output.
  372: 
  373:   # Case-sensitive (default)
  374:   status => {
  375:     type => 'string',
  376:     memberof => ['Draft', 'Published', 'Archived'] # Input 'draft' will fail - must match exact case
  377:   }
  378: 
  379:   # Case-insensitive
  380:   status => {
  381:     type => 'string',
  382:     memberof => ['Draft', 'Published', 'Archived'],
  383:     case_sensitive => 0 # Input 'draft', 'DRAFT', or 'DrAfT' will all pass
  384:   }
  385: 
  386:   country_code => {
  387:     type => 'string',
  388:     memberof => ['US', 'UK', 'CA', 'FR'],
  389:     case_sensitive => 0  # Accept 'us', 'US', 'Us', etc.
  390:   }
  391: 
  392: This flag has no effect on numeric types (C<integer>, C<number>, C<float>) as numbers
  393: do not have case.
  394: 
  395: =item * C<min>/C<minimum>
  396: 
  397: The minimum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  398: 
  399: =item * C<max>
  400: 
  401: The maximum length (for strings in characters not bytes), value (for numbers) or number of keys (for hashrefs).
  402: 
  403: =item * C<matches>
  404: 
  405: A regular expression that the parameter value must match.
  406: Checks all members of arrayrefs.
  407: 
  408: =item * C<nomatch>
  409: 
  410: A regular expression that the parameter value must not match.
  411: Checks all members of arrayrefs.
  412: 
  413: =item * C<position>
  414: 
  415: For routines and methods that take positional args,
  416: this integer value defines which position the argument will be in.
  417: If this is set for all arguments,
  418: C<validate_strict> will return a reference to an array, rather than a reference to a hash.
  419: 
  420: =item * C<regex>
  421: 
  422: Synonym of matches
  423: 
  424: =item * C<description>
  425: 
  426: The description of the rule
  427: 
  428: =item * C<callback>
  429: 
  430: A code reference to a subroutine that performs custom validation logic.
  431: The subroutine should accept the parameter value, the argument list and the schema as arguments and return true if the value is valid, false otherwise.
  432: 
  433: Use this to test more complex examples:
  434: 
  435:   my $schema = {
  436:     even_number => {
  437:       type => 'integer',
  438:       callback => sub { $_[0] % 2 == 0 }
  439:   };
  440: 
  441:   # Specify the arguments for a routine which has a second, optional argument, which, if given, must be less than or equal to the first
  442:   my $schema = {
  443:     first => {
  444:       type => 'integer'
  445:     }, second => {
  446:       type => 'integer',
  447:       optional => 1,
  448:       callback => sub {
  449:         my($value, $args) = @_;
  450: 	# The 'defined' is needed in case 'second' is evaluated before 'first'
  451: 	return (defined($args->{first}) && $value <= $args->{first}) ? 1 : 0
  452:       }
  453:     }
  454:   };
  455: 
  456: =item * C<optional>
  457: 
  458: A boolean value indicating whether the parameter is optional.
  459: If true, the parameter is not required.
  460: If false or omitted, the parameter is required.
  461: 
  462: It can be a reference to a code snippet that will return true or false,
  463: to determine if the parameter is optional or not.
  464: The code will be called with two arguments: the value of the parameter and hash ref of all parameters:
  465: 
  466:   my $schema = {
  467:     optional_field => {
  468:       type => 'string',
  469:       optional => sub {
  470:         my ($value, $all_params) = @_;
  471:         return $all_params->{make_optional} ? 1 : 0;
  472:       }
  473:     },
  474:     make_optional => { type => 'boolean' }
  475:   };
  476: 
  477:   my $result = validate_strict(schema => $schema, input => { make_optional => 1 });
  478: 
  479: If the parameter is not optional, it can be passed an undef value, which will not flag an error.
  480: This is by design.
  481: So this will not say that the required parameter 's' is missing:
  482: 
  483:     validate_strict(
  484:         schema => { s => { type => 'string' } },
  485:         input  => { s => undef },
  486:     );
  487: 
  488: =item * C<default>
  489: 
  490: Populate missing optional parameters with the specified value.
  491: Note that this value is not validated.
  492: 
  493:   username => {
  494:     type => 'string',
  495:     optional => 1,
  496:     default => 'guest'
  497:   }
  498: 
  499: =item * C<element_type>
  500: 
  501: Extends the validation to individual elements of arrays.
  502: 
  503:   tags => {
  504:     type => 'arrayref',
  505:     element_type => 'number',	# Float means the same
  506:     min => 1,	# this is the length of the array, not the min value for each of the numbers. For that, add a C<schema> rule
  507:     max => 5
  508:   }
  509: 
  510: =item * C<error_msg>
  511: 
  512: The custom error message to be used in the event of a validation failure.
  513: 
  514:   age => {
  515:     type => 'integer',
  516:     min => 18,
  517:     error_msg => 'You must be at least 18 years old'
  518:   }
  519: 
  520: =item * C<nullable>
  521: 
  522: Like optional,
  523: though this cannot be a coderef,
  524: only a flag.
  525: 
  526: =item * C<schema>
  527: 
  528: You can validate nested hashrefs and arrayrefs using the C<schema> property:
  529: 
  530:     my $schema = {
  531:         user => {	# 'user' is a hashref
  532:             type => 'hashref',
  533:             schema => {	# Specify what the elements of the hash should be
  534:                 name => { type => 'string' },
  535:                 age => { type => 'integer', min => 0 },
  536:                 hobbies => {	# 'hobbies' is an array ref that this user has
  537:                     type => 'arrayref',
  538:                     schema => { type => 'string' }, # Validate each hobby
  539:                     min => 1 # At least one hobby
  540:                 }
  541:             }
  542:         }, metadata => {
  543:             type => 'hashref',
  544:             schema => {
  545:                 created => { type => 'string' },
  546:                 tags => {
  547:                     type => 'arrayref',
  548:                     schema => {
  549:                         type => 'string',
  550:                         matches => qr/^[a-z]+$/	# Or you can say matches => '^[a-z]+$'
  551:                     }
  552:                 }
  553:             }
  554:         }
  555:     };
  556: 
  557: =item * C<validate>
  558: 
  559: A snippet of code that validates the input.
  560: It's passed the input arguments,
  561: and return a string containing a reason for rejection,
  562: or undef if it's allowed.
  563: 
  564:     my $schema = {
  565:       user => {
  566:         type => 'string',
  567: 	validate => sub {
  568: 	  if($_[0]->{'password'} eq 'bar') {
  569: 	    return undef;
  570: 	  }
  571: 	  return 'Invalid password, try again';
  572: 	}
  573:       }, password => {
  574:          type => 'string'
  575:       }
  576:     };
  577: 
  578: =item * C<transform>
  579: 
  580: A code reference to a subroutine that transforms/sanitizes the parameter value before validation.
  581: The subroutine should accept the parameter value as an argument and return the transformed value.
  582: The transformation is applied before any validation rules are checked, allowing you to normalize
  583: or clean data before it is validated.
  584: 
  585: Common use cases include trimming whitespace, normalizing case, formatting phone numbers,
  586: sanitizing user input, and converting between data formats.
  587: 
  588:   # Simple string transformations
  589:   username => {
  590:     type => 'string',
  591:     transform => sub { lc(trim($_[0])) },  # lowercase and trim
  592:     matches => qr/^[a-z0-9_]+$/
  593:   }
  594: 
  595:   email => {
  596:     type => 'string',
  597:     transform => sub { lc(trim($_[0])) },  # normalize email
  598:     matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  599:   }
  600: 
  601:   # Array transformations
  602:   tags => {
  603:     type => 'arrayref',
  604:     transform => sub { [map { lc($_) } @{$_[0]}] },  # lowercase all elements
  605:     element_type => 'string'
  606:   }
  607: 
  608:   keywords => {
  609:     type => 'arrayref',
  610:     transform => sub {
  611:       my @arr = map { lc(trim($_)) } @{$_[0]};
  612:       my %seen;
  613:       return [grep { !$seen{$_}++ } @arr];  # remove duplicates
  614:     }
  615:   }
  616: 
  617:   # Numeric transformations
  618:   quantity => {
  619:     type => 'integer',
  620:     transform => sub { int($_[0] + 0.5) },  # round to nearest integer
  621:     min => 1
  622:   }
  623: 
  624:   # Sanitization
  625:   slug => {
  626:     type => 'string',
  627:     transform => sub {
  628:       my $str = lc(trim($_[0]));
  629:       $str =~ s/[^\w\s-]//g;  # remove special characters
  630:       $str =~ s/\s+/-/g;      # replace spaces with hyphens
  631:       return $str;
  632:     },
  633:     matches => qr/^[a-z0-9-]+$/
  634:   }
  635: 
  636:   phone => {
  637:     type => 'string',
  638:     transform => sub {
  639:       my $str = $_[0];
  640:       $str =~ s/\D//g;  # remove all non-digits
  641:       return $str;
  642:     },
  643:     matches => qr/^\d{10}$/
  644:   }
  645: 
  646: The C<transform> function is applied to the value before any validation checks (C<min>/C<minimum>, C<max>,
  647: C<matches>, C<callback>, etc.), ensuring that validation rules are checked against the cleaned data.
  648: 
  649: Transformations work with all parameter types including nested structures:
  650: 
  651:   user => {
  652:     type => 'hashref',
  653:     schema => {
  654:       name => {
  655:         type => 'string',
  656:         transform => sub { trim($_[0]) }
  657:       }, email => {
  658:         type => 'string',
  659:         transform => sub { lc(trim($_[0])) }
  660:       }
  661:     }
  662:   }

  664: Transformations can also be defined in custom types for reusability:
  665: 
  666:   my $custom_types = {
  667:     email => {
  668:       type => 'string',
  669:       transform => sub { lc(trim($_[0])) },
  670:       matches => qr/^[\w\.\-]+@[\w\.\-]+\.\w+$/
  671:     }
  672:   };

  674: Note that the transformed value is what gets returned in the validated result and is what
  675: subsequent validation rules will check against. If a transformation might fail, ensure it
  676: handles edge cases appropriately.
  677: It is the responsibility of the transformer to ensure that the type of the returned value is correct,
  678: since that is what will be validated.
  679: 
  680: Many validators also allow a code ref to be passed so that you can create your own, conditional validation rule, e.g.:
  681: 
  682:   $schema = {
  683:     age => {
  684:       type => 'integer',
  685:       min => sub {

  688:       }
  689:     }
  690:   }

  696: =item * C<cross_validation>
  697: 
  698: A reference to a hash that defines validation rules that depend on more than one parameter.
  699: Cross-field validations are performed after all individual parameter validations have passed,
  700: allowing you to enforce business logic that requires checking relationships between different fields.
  701: 
  702: Each cross-validation rule is a key-value pair where the key is a descriptive name for the validation
  703: and the value is a code reference that accepts a hash reference of all validated parameters.
  704: The subroutine should return C<undef> if the validation passes, or an error message string if it fails.
  705: 
  706:   my $schema = {
  707:     password => { type => 'string', min => 8 },
  708:     password_confirm => { type => 'string' }
  709:   };
  710: 
  711:   my $cross_validation = {
  712:     passwords_match => sub {
  713:       my $params = shift;
  714:       return $params->{password} eq $params->{password_confirm}
  715:         ? undef : "Passwords don't match";
  716:     }
  717:   };
  718: 
  719:   my $validated = validate_strict(
  720:     schema => $schema,
  721:     input => $input,
  722:     cross_validation => $cross_validation
  723:   );
  724: 
  725: Common use cases include password confirmation, date range validation, numeric comparisons,
  726: and conditional requirements:
  727: 
  728:   # Date range validation
  729:   my $cross_validation = {
  730:     date_range_valid => sub {
  731:       my $params = shift;
  732:       return $params->{start_date} le $params->{end_date}
  733:         ? undef : "Start date must be before or equal to end date";
  734:     }
  735:   };
  736: 
  737:   # Price range validation
  738:   my $cross_validation = {
  739:     price_range_valid => sub {
  740:       my $params = shift;
  741:       return $params->{min_price} <= $params->{max_price}
  742:         ? undef : "Minimum price must be less than or equal to maximum price";
  743:     }
  744:   };

  746:   # Conditional required field
  747:   my $cross_validation = {
  748:     address_required_for_delivery => sub {
  749:       my $params = shift;
  750:       if ($params->{shipping_method} eq 'delivery' && !$params->{delivery_address}) {
  751:         return "Delivery address is required when shipping method is 'delivery'";
  752:       }
  753:       return undef;
  754:     }
  755:   };

  757: Multiple cross-validations can be defined in the same hash, and they are all checked in order.
  758: If any cross-validation fails, the function will C<croak> with the error message returned by the validation:
  759: 
  760:   my $cross_validation = {
  761:     passwords_match => sub {

  826: =item * C<relationships>
  827: 
  828: A reference to an array that defines validation rules based on relationships between parameters.
  829: Relationship validations are performed after all individual parameter validations have passed,
  830: but before cross-validations.
  831: 
  832: Each relationship is a hash reference with a C<type> field and additional fields depending on the type:
  833: 
  834: =over 4
  835: 
  836: =item * B<mutually_exclusive>
  837: 
  838: Parameters that cannot be specified together.
  839: 
  840:   relationships => [
  841:     {
  842:       type => 'mutually_exclusive',
  843:       params => ['file', 'content'],
  844:       description => 'Cannot specify both file and content'
  845:     }
  846:   ]
  847: 
  848: =item * B<required_group>
  849: 
  850: At least one parameter from the group must be specified.
  851: 
  852:   relationships => [
  853:     {
  854:       type => 'required_group',
  855:       params => ['id', 'name'],
  856:       logic => 'or',
  857:       description => 'Must specify either id or name'
  858:     }
  859:   ]
  860: 
  861: =item * B<conditional_requirement>
  862: 
  863: If one parameter is specified, another becomes required.
  864: 
  865:   relationships => [
  866:     {
  867:       type => 'conditional_requirement',
  868:       if => 'async',
  869:       then_required => 'callback',
  870:       description => 'When async is specified, callback is required'
  871:     }
  872:   ]
  873: 
  874: =item * B<dependency>
  875: 
  876: One parameter requires another to be present.
  877: 
  878:   relationships => [
  879:     {
  880:       type => 'dependency',
  881:       param => 'port',
  882:       requires => 'host',
  883:       description => 'port requires host to be specified'
  884:     }
  885:   ]
  886: 
  887: =item * B<value_constraint>
  888: 
  889: Specific value requirements between parameters.
  890: 
  891:   relationships => [
  892:     {
  893:       type => 'value_constraint',
  894:       if => 'ssl',
  895:       then => 'port',
  896:       operator => '==',
  897:       value => 443,
  898:       description => 'When ssl is specified, port must equal 443'
  899:     }
  900:   ]
  901: 
  902: =item * B<value_conditional>
  903: 
  904: Parameter required when another has a specific value.
  905: 
  906:   relationships => [
  907:     {
  908:       type => 'value_conditional',
  909:       if => 'mode',
  910:       equals => 'secure',
  911:       then_required => 'key',
  912:       description => "When mode equals 'secure', key is required"
  913:     }
  914:   ]
  915: 
  916: =back
  917: 
  918: If a parameter is optional and its value is C<undef>,
  919: validation will be skipped for that parameter.
  920: 
  921: If the validation fails, the function will C<croak> with an error message describing the validation failure.
  922: 
  923: If the validation is successful, the function will return a reference to a new hash containing the validated and (where applicable) coerced parameters.  Integer and number parameters will be coerced to their respective types.
  924: 
  925: The C<description> field is optional but recommended for clearer error messages.
  926: 
  927: =back
